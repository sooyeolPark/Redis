# Redis Cluster

데이터를 **여러 Master 노드에 슬롯 단위로 분산**하고, 각 Master마다 Replica를 두어 고가용성까지 갖춘 **수평 확장** 구성이다.

---

## 아키텍처 요약

- **슬롯 분산**  
  Redis Cluster는 키 공간을 **16384개 슬롯(0~16383)**으로 나눈다. 각 키는 해시 연산으로 하나의 슬롯에 매핑되고, 각 Master 노드가 슬롯 범위를 나누어 가진다. 예: Master 1은 0–5460, Master 2는 5461–10922, Master 3은 10923–16383. 클라이언트는 키를 기준으로 "이 키는 어느 노드에 있는지" 계산하여 **해당 Master에 직접** 요청을 보낸다.

- **클라이언트 동작**  
  클라이언트는 클러스터 내 아무 노드에나 접속할 수 있다. 요청한 키가 그 노드에 없으면(다른 슬롯이면) 서버가 "그 키는 X번 노드로 가라"는 **리다이렉트(MOVED/ASK)**를 반환하고, 클라이언트는 해당 노드로 다시 요청을 보낸다. 대부분의 클라이언트 라이브러리는 이 과정을 내부에서 처리해 준다.

- **Gossip 프로토콜**  
  Master 노드들은 **Gossip**으로 서로의 상태(어느 노드가 살아 있는지, 슬롯 할당이 어떻게 되어 있는지 등)를 주기적으로 교환한다. 이를 통해 클러스터 전체의 토폴로지와 슬롯 맵을 유지하고, 노드 추가·제거·장애 시에도 정보가 퍼져 나간다.

- **Replica와 고가용성**  
  각 Master에는 보통 하나 이상의 **Replica**가 붙는다. Replica는 Master의 슬롯 데이터를 복제한다. Master가 죽으면 해당 Replica가 **새 Master로 승격**되어 그 슬롯 범위를 이어받고, 클러스터는 계속 동작한다.

- **Sentinel과의 차이**  
  Sentinel은 "Master 한 개 + Replica들"을 모니터링하고 자동 페일오버만 담당한다. Cluster는 **데이터를 여러 Master에 나누어 저장(샤딩)**하고, 각 파트에 대해 Replica로 고가용성을 제공한다. 데이터량이 많거나 쓰기 부하가 클 때 수평 확장이 필요하면 Cluster를 사용하는 것이 적합하다.

## 클러스터의 주요 특성 (상세)

Redis 클러스터는 다음 세 가지 축으로 요약할 수 있다.

| 특성 | 설명 |
|------|------|
| **수평 확장 및 데이터 분산** | **16384 해시 슬롯 샤딩**을 사용한다. 전체 키 공간을 16384개의 슬롯으로 나누고, 이 슬롯들을 여러 마스터 노드에 나누어 할당한다. 노드를 늘리면 슬롯을 재분배하여 용량과 처리량을 확장할 수 있다. |
| **고가용성 및 자동 장애 조치** | **마스터-레플리카 자동 승격**을 지원한다. 각 마스터마다 레플리카를 두고, 마스터 장애 시 레플리카가 자동으로 마스터로 승격되어 서비스가 끊기지 않도록 한다. |
| **높은 성능 및 병렬 처리** | **여러 노드 동시 처리**가 가능하다. 슬롯별로 담당 노드가 나뉘어 있으므로, 서로 다른 슬롯에 대한 요청은 서로 다른 노드에서 병렬로 처리된다. |

## 키-슬롯 매핑 (16384 슬롯 샤딩)

클러스터에서 **어느 키가 어느 노드에 저장되는지**는 항상 다음 과정으로 결정된다.

### 계산 과정

1. **키**  
   클라이언트가 사용하는 키(예: `user:123`)가 입력이다. 해시 태그(`{...}`)를 사용하면 태그 안의 문자열만으로 슬롯이 계산되며, 같은 태그를 쓰는 키는 같은 슬롯(즉 같은 노드)에 모인다.

2. **CRC16 해시**  
   Redis는 해당 키(또는 해시 태그 부분)에 **CRC16** 해시를 적용한다.

3. **해시값 % 16384**  
   CRC16 결과를 **16384로 나눈 나머지**를 구한다. Redis 클러스터의 슬롯 번호는 0부터 16383까지이므로, 이 연산으로 **슬롯 번호**가 하나 정해진다.

4. **슬롯 번호 → 노드**  
   클러스터 구성에 따라 각 슬롯은 특정 **마스터 노드**에 할당되어 있다. 예를 들어 마스터 1은 슬롯 0–5460, 마스터 2는 5461–10922, 마스터 3은 10923–16383을 담당할 수 있다. 위에서 나온 슬롯 번호(예: 8452)가 속한 범위의 마스터가 해당 키를 저장하는 노드가 된다.

### 예시

- 키 `user:123` → CRC16 → 해시값 % 16384 → **슬롯 8452**  
- 슬롯 8452는 5461–10922 구간에 속하므로 **마스터 2**가 담당한다.  
- 따라서 `user:123`에 대한 읽기·쓰기는 최종적으로 **마스터 2**에서 처리된다.

이렇게 **16384 슬롯**과 **CRC16 % 16384** 규칙으로 데이터가 노드에 골고루 분산되며, 노드 수가 바뀌면 슬롯만 재분배하면 된다.

## 클러스터 아키텍처

### 노드 구성

- **마스터 노드**  
  슬롯 범위를 할당받아 해당 슬롯의 키에 대한 읽기·쓰기를 처리한다. 16384개 슬롯 전체가 여러 마스터에 나뉘어 할당된다(예: 3개 마스터면 각각 약 5461개 구간).

- **레플리카 노드**  
  각 마스터에 대해 하나 이상의 레플리카를 둘 수 있다. 레플리카는 마스터의 데이터를 비동기로 복제한다. 마스터 장애 시 **자동 장애 조치**로 레플리카가 마스터로 승격되어 같은 슬롯 범위를 이어받는다.

### 노드 간 통신: 가십(Gossip)

마스터 노드들은 **가십 프로토콜**로 서로의 상태를 주기적으로 교환한다. 노드 추가·제거, 슬롯 이동, 장애 감지 등이 이 정보를 바탕으로 이루어진다. 이를 통해 클러스터 전체의 토폴로지와 슬롯-노드 매핑이 유지된다.

### 클라이언트와 리다이렉션

클라이언트는 **아무 마스터나** 한 노드에 접속해 요청을 보낼 수 있다. 요청한 키의 슬롯을 해당 노드가 담당하지 않으면, 그 노드는 **리다이렉션**으로 “이 슬롯은 다른 노드가 담당한다”는 정보를 돌려준다. 클라이언트는 이 정보를 사용해 올바른 노드로 다시 요청을 보낸다. 대부분의 클러스터 지원 클라이언트 라이브러리는 이 리다이렉션을 자동으로 처리하고, 슬롯-노드 매핑을 캐시하여 이후 요청을 적절한 노드로 바로 보낸다.

## 클라이언트 리다이렉션 (MOVED)

클라이언트가 **잘못된 노드**에 명령을 보냈을 때의 흐름은 다음과 같다.

### 단계별 흐름 (예: SET user:123)

1. **클라이언트 → 마스터 1: SET user:123**  
   클라이언트가 먼저 접속한 노드(마스터 1)에 `SET user:123`을 보낸다.

2. **마스터 1: 슬롯 계산**  
   마스터 1은 `user:123`에 대해 CRC16 후 16384로 나눈 나머지를 계산해 **슬롯 8452**를 얻는다. 자신이 담당하는 슬롯 범위에 8452가 없음을 확인한다.

3. **마스터 1 → 클라이언트: MOVED 8452 127.0.0.1:7001**  
   마스터 1은 **MOVED** 응답을 반환한다. “슬롯 8452는 지금 `127.0.0.1:7001`(마스터 2)이 담당한다”는 의미이다. MOVED는 **영구적인** 슬롯 이동을 알리므로, 클라이언트는 슬롯-노드 매핑 캐시를 **갱신**해야 한다.

4. **클라이언트 → 마스터 2: SET user:123**  
   클라이언트는 MOVED에 적힌 주소(마스터 2)로 `SET user:123`을 다시 보낸다.

5. **마스터 2 → 클라이언트: OK**  
   마스터 2는 슬롯 8452를 담당하므로 명령을 수행하고 `OK`를 반환한다.

이 메커니즘으로 **데이터가 분산되어 있어도** 클라이언트는 최종적으로 올바른 노드와 통신하게 되며, 클라이언트 라이브러리가 MOVED를 처리하고 매핑을 캐시하면 개발자가 노드 주소를 직접 관리할 필요가 없다.

## 클러스터 재구성: 노드 추가·제거와 슬롯 마이그레이션

노드를 추가하거나 제거할 때는 **슬롯 재분배**와 **리다이렉션 타입(MOVED / ASK)** 이해가 필요하다.

### 1. 노드 추가 (Scale Out)

1. **새 노드 시작**  
   새로운 Redis 인스턴스를 클러스터 모드로 기동한다.

2. **클러스터 조인**  
   새 노드는 `CLUSTER MEET` 등으로 기존 클러스터에 참여하고, **가십 핸드셰이크**로 다른 노드들에게 자신을 알린다.

3. **빈 마스터 등록**  
   새 노드는 초기에 **슬롯 0개**를 가진 빈 마스터로 등록된다.

4. **슬롯 재분배 계획**  
   클러스터는 16384개 슬롯을 노드 수에 맞게 **균등 분산**하도록 재분배 계획을 세운다.

5. **슬롯 마이그레이션 (무중단)**  
   기존 노드에서 새 노드로 슬롯(그 안의 키들)이 **점진적으로** 이동한다. 이 과정은 **섹션 2**의 슬롯 마이그레이션 상태와 연동된다.

### 2. 슬롯 마이그레이션 상태

슬롯이 한 노드에서 다른 노드로 옮겨지는 **중간 단계**이다. 목표는 무중단 서비스이다.

- **타겟: IMPORTING**  
  슬롯을 **받는** 노드는 해당 슬롯에 대해 **IMPORTING** 상태가 된다.

- **소스: MIGRATING**  
  슬롯을 **내보내는** 노드는 해당 슬롯에 대해 **MIGRATING** 상태가 된다.

- **키 복사 및 점진적 이동**  
  소스 노드에서 타겟 노드로 키가 **점진적으로** 복사·이동한다.

- **마이그레이션 중 요청 처리**  
  - 해당 슬롯의 키에 대한 요청이 **이미 타겟으로 옮겨진 키**라면 → **ASK 리다이렉션**으로 클라이언트를 타겟으로 안내한다.  
  - **아직 소스에 있는 키**라면 → **소스 노드**가 그대로 처리한다.

- **이동 완료 후**  
  슬롯의 모든 키가 타겟으로 옮겨지면, 해당 슬롯의 소유권이 타겟으로 **영구 이전**되고, 이후에는 **MOVED**로 그 슬롯의 새 노드를 알린다.

### 3. 리다이렉션 타입: MOVED vs ASK

| 구분 | MOVED | ASK |
|------|--------|-----|
| **의미** | 해당 슬롯이 **영구적으로** 다른 노드로 이전되었음을 알린다. | 슬롯 **마이그레이션 중**에, 이번 요청한 키만 임시로 다른 노드(타겟)에 있음을 알린다. |
| **클라이언트 동작** | 슬롯-노드 매핑 **캐시를 갱신**한다. 이후 같은 슬롯에 대해서는 새 노드로 바로 요청한다. | 이번 요청만 **ASK에 적힌 노드**로 보내고, **캐시는 갱신하지 않는다**. 다음 요청은 기존 매핑대로 보낸다. |
| **발생 시점** | 슬롯 재분배가 끝난 뒤, 또는 클라이언트가 오래된 노드로 요청했을 때. | 슬롯이 MIGRATING/IMPORTING 상태일 때, 키가 이미 타겟으로 옮겨진 경우. |

### 4. 노드 제거 (Scale In)

1. **제거 대상 선정**  
   클러스터에서 빼낼 노드를 정한다. 마스터를 제거하려면 그 노드가 가진 **모든 슬롯**을 다른 노드로 옮겨야 한다.

2. **슬롯을 다른 노드로 이동**  
   제거 대상 노드의 슬롯들을 클러스터 내 다른 마스터들로 **마이그레이션**한다(위의 슬롯 마이그레이션과 동일한 절차).

3. **모든 슬롯 이동 완료**  
   제거 대상 노드의 슬롯이 모두 비면, 해당 노드는 더 이상 데이터를 갖지 않는다.

4. **노드 제거**  
   클러스터에서 해당 노드를 제거한다. 레플리카를 제거할 때는 슬롯 이동 없이 노드만 제거할 수 있다.

### 5. 리밸런싱

- **슬롯 분포 분석**  
  현재 클러스터에서 슬롯이 노드별로 **불균형**하게 쏠려 있는지 분석한다.

- **목표 슬롯 수**  
  이상적으로는 각 마스터가 **16384 / N**개(N = 마스터 수)의 슬롯을 갖도록 한다.

- **슬롯 이동 계획**  
  목표에 맞추기 위해 **최소 이동량**으로 슬롯 재분배 계획을 세운다.

- **점진적 재분배**  
  계획에 따라 슬롯을 **점진적으로** 옮기며, 이 과정은 **무중단**으로 수행된다.

---

## 요약

- Redis 클러스터는 **16384 해시 슬롯**으로 데이터를 샤딩하고, **CRC16(key) % 16384**로 키가 어느 슬롯(어느 노드)에 갈지 결정한다.  
- **마스터-레플리카**와 **가십**으로 토폴로지와 장애를 관리하며, **MOVED/ASK** 리다이렉션으로 클라이언트가 올바른 노드와 통신하게 한다.  
- **노드 추가·제거** 시에는 슬롯 마이그레이션과 리밸런싱을 통해 무중단으로 클러스터를 재구성할 수 있다.
