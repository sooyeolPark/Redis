# Redis 트랜잭션 (Transaction)

Redis 트랜잭션은 **MULTI**로 시작해 **EXEC**로 끝나는 구간 안의 여러 명령을 **한 번에 큐에 넣고, EXEC 시 순차적으로 한꺼번에 실행**하는 메커니즘이다. 원자성은 제한적으로만 지원되며, **WATCH**로 낙관적 잠금을 걸 수 있다. 관계형 DB의 트랜잭션과는 보장 수준이 다르므로 그 차이를 이해하고 사용해야 한다.

---

## 트랜잭션의 특성

### 1. 원자성 (Atomicity) — All or Nothing

- **의미:** MULTI~EXEC 블록 안의 명령들이 **한 단위**처럼 처리된다. EXEC가 호출되면 큐에 쌓인 명령이 **차례대로 모두** 실행되고, 그 사이에 다른 클라이언트의 명령이 끼어들지 않는다.
- **부분적 지원, 롤백 없음:** Redis는 **실행된 명령을 되돌리는 롤백(rollback)**을 제공하지 않는다. 예를 들어 명령 1이 성공한 뒤 명령 2에서 오류가 나도, 명령 1의 결과는 그대로 남는다. “중간 실패 시 이전 명령까지 취소”가 아니라 “중간 실패 시 그 시점까지 실행된 것은 유지”이다. 따라서 **진정한 All-or-Nothing**이라기보다는 **“한 클라이언트의 명령들을 연속으로 실행하는 배치”**에 가깝다.
- **실행 시점:** MULTI 이후 보낸 명령은 **즉시 실행되지 않고 큐에만 쌓인다.** EXEC를 보내는 순간 그 큐가 **한 번에 순차 실행**된다.

### 2. 명령 일괄 실행 및 순차 처리

- **명령 일괄 실행:** MULTI 다음에 오는 명령들은 **큐에만 적재**되고, EXEC가 호출될 때 비로소 **한꺼번에** 실행된다.
- **순차 처리:** 큐에 들어간 순서대로 **하나씩** 실행된다. 명령 1 → 명령 2 → … → 명령 N 순서가 보장된다.
- **다른 클라이언트 끼어들기 방지:** Redis는 단일 스레드로 명령을 처리하므로, 한 클라이언트가 EXEC로 “큐 전체”를 실행하는 동안 **다른 클라이언트의 명령은 그 블록이 끝난 뒤에만** 실행된다. 즉, MULTI~EXEC 블록 단위로 **격리**된 실행이 이루어진다.

### 3. WATCH — 낙관적 잠금 (Optimistic Locking)

- **동작:** `WATCH key [key ...]`로 **키(들)**를 감시할 수 있다. EXEC를 호출하기 **전에** 감시 중인 키가 **다른 클라이언트에 의해 변경**되었으면, EXEC 시 해당 트랜잭션은 **실행되지 않고** nil(또는 실패 응답)을 반환한다.
- **변경 감지:** WATCH한 키에 SET, INCR, DEL 등이 실행되면 “변경됨”으로 표시된다. EXEC 시점에 이 표시가 있으면 **조건부 실행**이 실패한다.
- **조건부 실행:** “내가 읽은 값이 그대로일 때만 실행”하고 싶을 때 WATCH를 쓰면 된다. EXEC가 실패하면 클라이언트는 다시 WATCH부터 MULTI~EXEC를 반복할 수 있다. **락을 걸지 않고** “변경됐으면 재시도”하는 **낙관적 잠금** 패턴이다.
- **UNWATCH:** WATCH를 취소한다. MULTI 없이 호출 가능하다.

## MULTI / EXEC 흐름

1. **클라이언트가 MULTI 전송**  
   Redis 서버는 “이제 오는 명령은 실행하지 말고 **큐에만 넣어라**”는 상태로 전환한다.

2. **명령 1, 2, …, N 전송 (큐잉)**  
   클라이언트가 보내는 각 명령은 **즉시 실행되지 않고** 트랜잭션 큐에 **순서대로** 쌓인다. 서버는 각 명령에 대해 `QUEUED`를 반환한다.

3. **클라이언트가 EXEC 전송**  
   서버는 큐에 쌓인 **모든 명령**을 **앞에서부터 차례대로** 실행한다. 이 구간 동안 다른 클라이언트의 명령은 처리되지 않는다.

4. **명령 1 실행 → 명령 2 실행 → … → 명령 N 실행**  
   Redis 서버가 큐에 들어 있던 명령을 **순차적으로** 수행한다.

5. **결과 배열 반환**  
   실행이 끝나면 **각 명령의 반환값이 순서대로 담긴 배열**을 클라이언트에게 한 번에 반환한다. 예: `[결과1, 결과2, ..., 결과N]`. 중간에 실행 오류가 난 명령이 있으면 해당 위치에 에러 객체가 들어가고, Redis 버전/클라이언트에 따라 그 뒤 명령 실행 여부가 달라질 수 있다.

**정리:** MULTI → (명령 큐잉) → EXEC → (한 번에 순차 실행) → (결과 배열 반환) 순서로 동작한다.

## DISCARD

- **동작:** `DISCARD`는 MULTI 이후 쌓아 둔 **트랜잭션 큐를 버리고** 트랜잭션 모드를 종료한다. EXEC를 하지 않고 “이번에는 실행하지 않겠다”고 취소할 때 사용한다.
- **WATCH와의 관계:** DISCARD가 호출되면 **자동으로 UNWATCH**가 수행된 것처럼 WATCH도 해제된다.

## ACID 관점에서의 Redis 트랜잭션

관계형 DB에서 말하는 ACID와 Redis 트랜잭션은 **완전히 같지 않다**. 각 속성을 Redis 기준으로 정리하면 다음과 같다.

| 속성 | Redis에서의 특성 |
|------|------------------|
| **원자성 (Atomicity)** | **부분적 지원. 롤백 없음.** MULTI~EXEC 블록은 한꺼번에 실행되지만, 실행된 명령을 되돌리는 기능은 없다. 중간에 실패해도 이미 실행된 명령은 그대로 남는다. |
| **일관성 (Consistency)** | **애플리케이션 책임.** Redis가 제약 조건(외래키, 유니크 등)을 강제하지 않으므로, “데이터가 항상 유효한 상태”를 지키는 것은 **애플리케이션 로직**과 트랜잭션 설계에 맡겨진다. |
| **격리성 (Isolation)** | **완전 지원. 단일 스레드.** 한 번에 하나의 명령(또는 EXEC로 묶인 명령 묶음)만 처리되므로, 다른 클라이언트 명령이 그 사이에 끼어들지 않는다. |
| **지속성 (Durability)** | **설정 의존. AOF/RDB.** 트랜잭션 결과가 디스크에 남는지는 **영속성 설정**(RDB 스냅샷, AOF 등)에 따라 달라진다. 영속성을 끄면 재시작 시 데이터가 사라진다. |

따라서 Redis 트랜잭션은 “명령 배치 + 격리 + (선택) WATCH”로 이해하고, 롤백·강한 일관성·지속성은 설정과 애플리케이션 설계로 보완해야 한다.

## 사용 시 유의사항

- **롤백 불가:** 실행된 명령은 취소할 수 없다. “되돌려야 하는” 연산이 있으면 애플리케이션에서 보상 로직(역연산, 보상 트랜잭션 등)을 구현해야 한다.
- **스크립트와의 관계:** 복잡한 조건·반복이 필요하면 **Lua 스크립트**(EVAL)가 더 적합할 수 있다. 스크립트는 서버에서 원자적으로 실행되며, 트랜잭션보다 로직을 표현하기 쉽다.
- **WATCH 실패 시:** EXEC가 nil(또는 실패)로 끝나면 WATCH된 키가 중간에 변경된 것이다. 재시도할 때는 **새로 값을 읽고**, 다시 WATCH한 뒤 MULTI~EXEC를 수행하는 패턴을 사용한다.
