# Redis Hash 타입

Redis Hash는 **하나의 키 아래에 여러 필드-값 쌍**을 저장하는 자료구조이다. 객체(사용자, 상품 등)를 하나의 키로 표현할 수 있고, 필드 단위로 조회·수정이 가능하며, 작은 Hash는 압축 인코딩으로 메모리를 절약한다.

---

## Hash의 주요 특징

### 1. 필드-값 쌍 저장 (Field-Value Pairs)

Hash는 **한 Redis 키 안에** 여러 개의 **필드(field)**와 **값(value)** 쌍을 가진다.

- **키 내부의 Key-Value:** Redis 키(`user:100`) 하나가 Hash이고, 그 안에 `name`, `email`, `age` 같은 필드가 각각 값에 대응한다. 프로그래밍의 객체(Object)나 맵(Map)과 비슷한 구조이다.
- **단일 키로 여러 속성 관리:** 사용자 한 명, 상품 한 개를 하나의 키로 표현하고, 속성만 필드로 나누어 저장할 수 있다.

### 2. 객체 표현에 최적화

구조화된 데이터를 **객체 단위**로 다루기에 적합하다.

- **사용자 정보, 상품 정보 등:** `user:123`에 `name`, `email`, `age`, `last_login`을 필드로 두거나, `product:456`에 `name`, `price`, `stock`을 두는 식으로 활용한다.
- **JSON 전체를 String으로 저장하는 것과의 차이:** String 하나에 JSON을 넣으면 한 필드만 바꿔도 전체를 읽고 다시 써야 한다. Hash는 **필드 단위**로 읽고 쓰므로 갱신이 효율적이다.

### 3. 메모리 효율적 (Memory Efficient)

작은 Hash는 Redis가 **압축 인코딩**을 사용하여 메모리를 절약한다.

- **압축 인코딩 지원:** 필드 수와 값 길이가 일정 기준 이하이면 listpack(또는 구 버전 ziplist)으로 저장되어, 필드-값을 각각 별도 Redis 키로 둔 경우보다 **오버헤드가 적다**.
- **기준:** `hash-max-listpack-entries`, `hash-max-listpack-value` 등으로 조정 가능하다. 기준을 넘으면 일반 해시 테이블로 전환된다.
- **활용:** 사용자/상품처럼 “작은 객체”가 많을 때 Hash를 쓰면 메모리 사용량을 줄이는 데 도움이 된다.

### 4. 부분 업데이트 가능 (Partial Update)

**필드별로 개별 접근**할 수 있어, 일부만 수정·조회할 수 있다.

- **필드 단위 연산:** HGET, HSET, HINCRBY, HDEL 등으로 **특정 필드만** 읽고 쓰고 지울 수 있다. 다른 필드는 건드리지 않는다.
- **장점:** 네트워크 전송량과 직렬화 비용이 줄고, 동시에 여러 클라이언트가 서로 다른 필드를 수정해도 충돌 범위가 좁다. 예: 이름만 바꿀 때 `HSET user:1 name "새이름"`만 보내면 된다.

## Hash 기본 명령어 (HSET, HGET)

### HSET (필드 설정)

- **동작:** `HSET key field value [field value ...]`는 해당 키의 Hash에 **필드-값**을 설정한다. 필드가 없으면 새로 만들고, 있으면 **값을 덮어쓴다**.
- **단일/다중 필드 설정:** 필드-값 쌍을 한 개만 주면 한 필드만 설정하고, 여러 쌍을 주면 **한 번에 여러 필드**를 설정할 수 있다. 예: `HSET user:1 name "Park" email "park@example.com"`.
- **반환값:** Redis 4.0.0 이상에서는 **새로 필드가 추가된 개수**를 반환한다. 기존 필드 값만 바꾼 경우 0이다. (이전 버전은 OK 등 다를 수 있다.)
- **시간 복잡도:** 필드 하나당 **O(1)**.

### HGET (필드 조회)

- **동작:** `HGET key field`는 해당 Hash에서 **지정한 필드의 값** 하나만 가져온다.
- **단일 필드 값 반환:** 문자열 한 개가 반환된다. 필드가 없으면 nil을 반환한다.
- **시간 복잡도:** **O(1)**. 필드 개수와 관계없이 일정하다.

키가 없거나 해당 키가 Hash 타입이 아니면 적절히 nil 또는 오류가 반환된다.

## 다중 필드 작업 (HMSET, HMGET)

### HMSET (여러 필드 한 번에 설정)

- **동작:** `HMSET key field value [field value ...]`는 **여러 필드-값**을 한 번에 설정한다. 동작 자체는 HSET에 여러 쌍을 넘기는 것과 같다.
- **HSET으로 대체됨:** Redis 4.0.0부터 HSET이 여러 필드-값을 받을 수 있으므로, **새 코드에서는 HSET을 쓰는 것이 권장**된다.
- **하위 호환성 유지:** HMSET은 deprecated이지만 제거되지 않고 동작한다. 기존 스크립트·클라이언트와의 호환을 위해 남아 있다.

### HMGET (여러 필드 한 번에 조회)

- **동작:** `HMGET key field [field ...]`는 지정한 **여러 필드**의 값을 한 번에 가져온다.
- **배열로 값 반환:** 요청한 **필드 순서대로** 값이 배열로 반환된다. 필드가 없으면 해당 자리는 nil이다.
- **필드 순서 보존:** 인자로 준 필드 순서와 반환 배열의 순서가 같아서, 클라이언트에서 필드와 값을 쉽게 매핑할 수 있다.
- **시간 복잡도:** 요청한 필드 개수에 비례한다. 필드마다 O(1)이므로 N개면 O(N)이다. HGET을 N번 호출하는 것보다 **한 번의 왕복**으로 처리할 수 있어 RTT가 줄어든다.

## 필드 증가 명령어 (HINCRBY, HINCRBYFLOAT)

Hash 필드의 **숫자 값**을 원자적으로 증가·감소시키는 명령이다.

### HINCRBY (정수 증가)

- **동작:** `HINCRBY key field increment`는 해당 필드의 값을 **정수만큼** 증가시킨다. 필드가 없으면 0으로 간주한 뒤 증가시킨다. `increment`가 음수면 감소한다.
- **64비트 정수 범위:** 연산 결과는 64비트 부호 있는 정수 범위(-2^63 ~ 2^63-1) 안이어야 한다. 벗어나면 오류가 발생한다.
- **활용:** 조회수, 재고 수량, 좋아요 수 등 **정수 카운터**를 필드로 둘 때 사용한다.

### HINCRBYFLOAT (실수 증가)

- **동작:** `HINCRBYFLOAT key field increment`는 해당 필드의 값을 **부동소수점만큼** 증가시킨다. 필드가 없으면 0.0으로 간주한다.
- **배정밀도 부동소수점:** IEEE 754 배정밀도(double)를 사용하여 소수 연산이 가능하다.
- **활용:** 평점 합계, 점수, 비율 등 **실수 누적**이 필요할 때 사용한다.

### 원자적 연산 (Atomic Operation)

- **원자성:** HINCRBY와 HINCRBYFLOAT는 **원자적**으로 실행된다. 여러 클라이언트가 동시에 같은 필드를 증가해도 연산이 쪼개지지 않고, **레이스 컨디션 없이** 올바른 결과가 유지된다.
- **동시성:** 락 없이 카운터·집계 필드를 안전하게 갱신할 수 있어, 조회수·재고·평점 집계 등에 적합하다.

## Hash 정보 조회 (HEXISTS, HKEYS, HVALS)

### HEXISTS (필드 존재 확인)

- **동작:** `HEXISTS key field`는 해당 Hash에 **지정한 필드가 있는지** 확인한다.
- **반환값:** 있으면 1, 없으면 0을 반환한다.
- **시간 복잡도:** **O(1)**. **빠른 체크**에 적합하다. 필드가 있을 때만 HGET을 호출하는 식으로 사용할 수 있다.

### HKEYS (모든 필드명)

- **동작:** `HKEYS key`는 해당 Hash의 **모든 필드 이름**을 배열로 반환한다.
- **시간 복잡도:** **O(N)**. N은 Hash의 필드 개수이다. 필드가 많을수록 비용이 커진다.
- **큰 Hash 주의:** 필드가 수천~수만 개인 Hash에서 HKEYS를 자주 호출하면 메모리와 CPU 부담이 크고, 다른 요청을 지연시킬 수 있다. 가능하면 필드 이름을 이미 알고 있는 경우 HGET/HEXISTS를 쓰는 것이 좋다.

### HVALS (모든 값)

- **동작:** `HVALS key`는 해당 Hash의 **모든 값**을 배열로 반환한다. 필드명은 반환되지 않는다.
- **시간 복잡도:** **O(N)**. HKEYS와 마찬가지로 필드 개수에 비례한다.
- **큰 Hash 주의:** HKEYS와 동일한 이유로, 큰 Hash에서는 사용을 자제하거나 HSCAN으로 순회하는 것을 고려한다.

## HGETALL, HDEL, HSTRLEN

### HGETALL (전체 필드-값 조회)

- **동작:** `HGETALL key`는 해당 Hash의 **모든 필드와 값**을 [field1, value1, field2, value2, ...] 형태의 배열로 반환한다.
- **시간 복잡도:** **O(N)**. 객체 전체를 한 번에 가져올 때 편하지만, 필드가 많으면 부담이 크다. **큰 Hash에서는 HGET/HMGET으로 필요한 필드만** 가져오는 것이 좋다.

### HDEL (필드 삭제)

- **동작:** `HDEL key field [field ...]`는 지정한 **필드들을 Hash에서 제거**한다. 없는 필드는 무시된다.
- **반환값:** 실제로 삭제된 필드 개수를 반환한다.
- **마지막 필드 삭제 시:** Hash에 필드가 하나도 없으면 Redis가 해당 키를 삭제한다.

### HSTRLEN (필드 값 길이)

- **동작:** `HSTRLEN key field`는 해당 **필드 값의 바이트 길이**를 반환한다. 필드가 없으면 0을 반환한다.
- **활용:** 값이 너무 큰지 확인하거나, 저장 전 검증할 때 사용할 수 있다.

## Hash vs String 선택


| 관점          | Hash                        | String (JSON 등)          |
| ----------- | --------------------------- | ------------------------ |
| **구조**      | 필드-값 쌍, 필드 단위 접근            | 한 키에 문자열 하나 (예: JSON)    |
| **부분 갱신**   | 필드 단위로 가능 (HSET, HINCRBY 등) | 전체를 읽고 수정 후 다시 SET       |
| **메모리**     | 작은 객체는 압축 인코딩으로 절약          | 키 오버헤드가 필드마다 있음 (필드별 키면) |
| **한 번에 조회** | HGETALL로 전체, 또는 HMGET으로 일부  | GET 한 번으로 전체만 가능         |


- **객체(엔티티) 단위로 여러 속성을 다루고, 일부만 자주 바꿀 때** → Hash가 유리하다.
- **항상 전체를 통째로 읽고 쓸 때**, 또는 **이미 JSON으로 직렬화된 문자열**을 그대로 캐시할 때 → String도 충분할 수 있다.

---

## 사용 시 유의사항

- **필드 개수:** 필드가 매우 많아지면 HKEYS, HVALS, HGETALL은 O(N)으로 부담이 커진다. 필요한 필드만 HGET/HMGET으로 가져오거나, HSCAN으로 순회하는 패턴을 고려한다.
- **값 크기:** Hash는 “작은 객체”에 최적화되어 있다. 필드 값이 매우 크면 String 타입 하나로 두는 편이 나을 수 있다.
- **타입 일관성:** 한 키는 하나의 타입만 가진다. 같은 키에 HSET과 SET을 섞어 쓰면 안 되고, 기존에 String인 키에 HSET을 하면 오류가 발생한다.
