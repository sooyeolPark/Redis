# Redis Stream

Redis Stream은 **추가 전용 로그(append-only log)** 형태의 메시지 스트림이다. 메시지는 **소비해도 삭제되지 않고** ID로 순서가 보장되며, **Consumer Group**으로 여러 소비자가 분산 처리하고 **XACK**로 처리 완료를 확인할 수 있다. Pub/Sub이나 List 기반 큐보다 **메시지 보존·재처리·분산 처리**에 유리하다.

---

## Stream의 특징

### 1. 메시지 보존 (Message Preservation)

- **소비해도 삭제되지 않음:** XREAD나 XREADGROUP으로 메시지를 읽어도 스트림에서 **자동으로 삭제되지 않는다**. 로그처럼 계속 남아 있어, 같은 메시지를 다시 읽거나 다른 소비자가 읽을 수 있다.
- **의미:** “한 번 읽으면 사라지는” List의 RPOP/LPOP과 다르다. 장애 복구, 재처리, 감사·디버깅에 유리하다.

### 2. 다중 소비자 (Multiple Consumers)

- **여러 소비자가 독립적으로 읽기:** 스트림 하나에 **여러 클라이언트**가 동시에 접근할 수 있다. 각자 다른 ID부터 읽거나, Consumer Group에 속해 메시지를 나누어 받을 수 있다.
- **의미:** “한 메시지를 한 소비자만” 처리하려면 Consumer Group을 쓰고, “모든 소비자가 전체 메시지”를 보려면 일반 XREAD를 쓰면 된다.

### 3. 시간순 정렬 (Time-ordered Sorting)

- **ID 기반 순서 보장:** 각 메시지는 **고유 ID**를 가지며, ID는 보통 **타임스탬프-시퀀스** 형태로 생성된다. 스트림 안에서 **ID 순서**가 곧 시간 순서가 되어, 범위 조회·재생이 쉽다.
- **의미:** XRANGE로 “이 ID ~ 이 ID” 구간만 조회하거나, 과거 메시지를 다시 읽을 수 있다.

### 4. 재처리 가능 (Re-processing Possible)

- **과거 메시지 다시 읽기:** 메시지가 보존되므로 **특정 ID 이후** 또는 **ID 구간**을 지정해 다시 읽을 수 있다. Consumer Group에서는 ACK하지 않은 메시지는 PEL에 남아 **XCLAIM**으로 다른 소비자가 재처리할 수 있다.
- **의미:** 처리 실패·타임아웃 시 같은 메시지를 다시 처리하는 패턴을 구현하기 좋다.

---

## 기본 명령: XADD, XREAD, XRANGE

### XADD (메시지 추가)

- **문법:** `XADD stream [MAXLEN len] * field value [field value ...]`
- **동작:** 지정한 스트림에 **새 메시지(엔트리)**를 추가한다. `*`는 “ID 자동 생성”을 의미한다.
- **자동 ID 생성 (타임스탬프-시퀀스):** Redis가 **밀리초 타임스탬프 + 시퀀스** 형태의 유일한 ID를 부여한다. 예: `1699123456789-0`. 같은 밀리초에 여러 개가 들어오면 시퀀스만 증가한다.
- **MAXLEN:** `MAXLEN len`을 주면 스트림 길이가 len을 넘을 때 **오래된 엔트리부터 제거**한다. 메모리와 보존 기간을 제한할 때 쓴다. 예: `XADD stream MAXLEN 1000 * a 1`.

### XREAD (메시지 읽기)

- **문법:** `XREAD [COUNT n] [BLOCK ms] STREAMS stream [stream ...] id [id ...]`
- **동작:** 지정한 스트림에서 **주어진 ID 이후**에 도착한 메시지를 읽는다. ID `0`이면 “가장 오래된 메시지부터”이다.
- **특정 ID 이후 메시지 조회:** 이전에 받은 마지막 ID를 넘기면, 그 다음 메시지들만 받을 수 있다. 증분 읽기에 적합하다.
- **BLOCK:** `BLOCK ms`를 주면 새 메시지가 없을 때 **최대 ms 밀리초**까지 대기한다. 새 메시지가 오면 즉시 반환한다. 실시간 소비에 쓴다.
- **특징:** Consumer Group을 쓰지 않으면 **같은 스트림을 읽는 모든 클라이언트가 동일한 메시지**를 각자 읽는다(브로드캐스트에 가깝다).

### XRANGE (범위 조회)

- **문법:** `XRANGE stream start end [COUNT n]`
- **동작:** 스트림에서 **시작 ID ~ 끝 ID** 구간의 메시지를 **ID 오름차순**으로 반환한다. `-`는 최소 ID, `+`는 최대 ID로 쓴다. `XRANGE stream - +`는 전체 조회이다.
- **시작~끝 ID 범위:** 특정 시간대나 ID 구간만 조회할 때 사용한다.
- **XREVRANGE:** `XREVRANGE stream end start`는 **ID 내림차순**(최신 먼저)으로 같은 구간을 조회한다.

## Pub/Sub vs Stream

| 구분 | Pub/Sub | Stream |
|------|---------|--------|
| **메시지 저장** | **저장하지 않음.** 발행 시점에 구독자에게만 전달되고, 구독자가 없으면 유실된다. | **저장함.** append-only 로그에 쌓이며, 소비 후에도 남는다. |
| **전달 방식** | **실시간 브로드캐스트.** 모든 구독자가 동일 메시지를 받는다. | **스트림에서 읽기.** 일반 XREAD는 “누가 읽든” 같은 메시지를 읽을 수 있고, Consumer Group이면 그룹 내에서 한 메시지를 한 소비자만 받는다. |
| **구조** | **단순.** 채널·패턴 구독만 있으면 된다. | **Consumer Group, ACK, PEL, XCLAIM** 등으로 신뢰성·분산 처리를 지원한다. |
| **ACK / 재처리** | 없음. 전달만 하고 끝. | **XACK**로 처리 완료 표시, **PEL**에 미완료 메시지 관리, **XCLAIM**으로 재처리 가능. |

**정리:** “실시간 알림, 유실 허용”이면 Pub/Sub, “메시지 보존·재처리·분산 처리”가 필요하면 Stream을 쓴다.

## Stream의 장점 vs List의 한계

List(RPUSH/LPOP 등)로 메시지 큐를 만들 때의 한계와 Stream이 어떻게 보완하는지 정리한다.

| 관점 | Stream | List |
|------|--------|------|
| **소비 후 메시지** | **보존된다.** 읽어도 스트림에 남아 있어 재조회·재처리가 가능하다. | **RPOP/LPOP하면 삭제된다.** 한 번 꺼내면 다른 소비자나 재처리를 위해 다시 쓸 수 없다. |
| **여러 소비자** | **Consumer Group**으로 그룹 내에서 메시지를 **분배**할 수 있다. 한 메시지는 그룹 안 한 소비자만 받는다. | 한 메시지를 여러 소비자가 “각자 한 번씩” 처리하려면 애플리케이션에서 복잡하게 나눠야 하고, RPOP하면 한 쪽만 가져가므로 구조상 불리하다. |
| **처리 완료 / 실패** | **XACK**로 “처리 완료”를 알릴 수 있고, ACK 안 된 메시지는 **PEL**에 남아 **XCLAIM**으로 다른 소비자가 재처리할 수 있다. | POP 후 처리 중 실패하면 **메시지가 유실**된다. 별도 “미완료 큐”를 앱에서 구현하지 않으면 복구가 어렵다. |
| **ID / 범위 조회** | **ID 기반**이라 **XRANGE** 등으로 구간·과거 메시지를 조회할 수 있다. | List에는 메시지 ID가 없어 **ID나 구간으로 조회**할 수 없다. |

**정리:** 메시지 큐·이벤트 로그처럼 “보존·분산·재처리”가 중요하면 Stream이 적합하고, “한 번 꺼내면 끝”인 단순 큐면 List도 가능하다.

## Consumer Group vs 일반 XREAD

### 일반 XREAD

- **동작:** 스트림 이름과 “어디서부터 읽을지(ID)”만 지정해 읽는다. **Consumer Group과 무관**하다.
- **결과:** **같은 스트림을 읽는 모든 클라이언트**가 **동일한 메시지 전체**를 각자 읽는다. 예: 메시지 1, 2, 3이 있으면 Consumer A·B·C 모두 1, 2, 3을 받는다.
- **용도:** 브로드캐스트(여러 구독자가 같은 이벤트를 각자 처리), 단일 소비자, 또는 “전체 로그를 복사”하는 패턴에 쓴다.

### Consumer Group

- **동작:** `XGROUP CREATE`로 그룹을 만든 뒤, **XREADGROUP GROUP 그룹명 소비자명**으로 읽는다. `>`는 “아직 이 그룹에 할당되지 않은 새 메시지만” 의미한다.
- **결과:** **그룹 안에서** 메시지가 **소비자들에게 나뉘어** 전달된다. 한 메시지는 그룹 내 **한 소비자만** 받는다. 예: Consumer A는 1, B는 2, C는 3을 받는 식으로 분배된다.
- **용도:** “한 메시지를 한 번만 처리”하는 **분산 처리·로드 밸런싱**에 적합하다. 여러 워커가 같은 스트림을 나누어 처리할 때 사용한다.

**정리:** “모두가 같은 메시지를 본다” → 일반 XREAD. “그룹 단위로 메시지를 나눠 처리한다” → Consumer Group + XREADGROUP.

## Consumer Group과 PEL (Pending Entries List)

Consumer Group으로 메시지를 읽으면, “할당은 됐지만 아직 ACK하지 않은” 메시지는 **PEL(Pending Entries List)**에 남는다.

### 흐름 요약

1. **Producer:** `XADD`로 스트림에 메시지를 추가한다.
2. **Consumer:** `XREADGROUP GROUP ... STREAMS stream >`으로 **그룹에 할당되지 않은 새 메시지**를 요청한다.
3. **Stream:** 해당 메시지를 해당 소비자에게 **전달**하고, 그 메시지 ID를 **PEL에 기록**한다(해당 그룹·소비자 기준으로 “처리 중” 상태가 됨).
4. **Consumer:** 메시지를 처리한다.

### 처리 성공 시

- **Consumer:** `XACK stream 그룹명 메시지ID`로 **처리 완료**를 알린다.
- **Stream:** PEL에서 해당 메시지 ID를 **제거**한다. 해당 메시지는 “그룹 기준으로 처리 완료”로 간주된다.

### 처리 실패 / 타임아웃 시

- **메시지는 PEL에 남는다.** ACK가 없으므로 “아직 처리 완료되지 않음”으로 유지된다.
- **다른 Consumer가 재처리:** `XCLAIM`으로 해당 메시지의 **소유권을 다른 소비자에게 넘길** 수 있다. (예: 원래 소비자 타임아웃 후 다른 워커가 인계받아 재처리.)
- **XPENDING**으로 그룹·소비자별 미완료 메시지 목록을 조회할 수 있다.

**정리:** PEL + XACK + XCLAIM으로 “한 번만 처리”와 “실패 시 재처리”를 조합할 수 있다.

## 사용 시 유의사항

- **Consumer Group 생성:** `XGROUP CREATE stream 그룹명 id`에서 `id`가 `0`이면 “스트림 기존 메시지부터” 그룹에 포함하고, `$`이면 “이후에 들어오는 메시지만” 그룹에서 읽는다. 이미 데이터가 쌓인 스트림에 `$`로 만들면 기존 메시지는 그룹에서 읽지 못한다.
- **XLEN / MAXLEN:** 스트림 크기가 커지면 메모리와 성능에 영향을 준다. 필요하면 `XADD ... MAXLEN`으로 길이를 제한한다.
- **블로킹:** `XREAD BLOCK` 또는 `XREADGROUP`에 블로킹 옵션을 주면 연결이 대기 상태가 되므로, 클라이언트/스레드 모델에 맞게 사용한다.
