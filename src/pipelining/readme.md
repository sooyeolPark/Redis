# Redis 파이프라이닝 (Pipelining)

파이프라이닝은 클라이언트가 **여러 명령을 응답을 기다리지 않고 연속으로 보내고**, 서버는 그대로 **순서대로 실행한 뒤** 응답을 **한꺼번에** 돌려주는 방식이다. **네트워크 왕복(RTT) 횟수를 줄여** 지연을 낮추고 처리량을 높인다. 트랜잭션(MULTI/EXEC)과는 목적이 다르다. 트랜잭션은 “한 덩어리로 실행·격리”이고, 파이프라이닝은 “네트워크 최적화”에 가깝다.

---

## RTT와 명령-응답 흐름

클라이언트가 한 개의 명령을 보내고 그 응답을 받기까지의 과정은 다음과 같다.

1. **명령 전송:** 클라이언트가 Redis 서버로 명령을 보낸다.
2. **네트워크 구간(지연):** 명령이 네트워크를 지나 서버에 도달할 때까지 **지연**이 발생한다. 이 구간이 **RTT(왕복 시간)의 절반**을 차지한다.
3. **Redis 서버:** 서버에 명령이 도착하면 **명령을 처리**하고 **응답을 생성**한다.
4. **응답 전송:** 서버가 생성한 응답을 클라이언트로 보낸다.
5. **네트워크 구간(지연):** 응답이 네트워크를 지나 클라이언트에 도달할 때까지 다시 **지연**이 발생한다. 이 구간이 RTT의 **나머지 절반**을 차지한다.
6. **클라이언트:** 응답을 수신한다.

**RTT(Round-Trip Time)**는 “클라이언트 → 서버 → 클라이언트” 한 번 왕복하는 데 걸리는 시간이다. 명령을 N번 보내고 각각 응답을 기다리면 **RTT가 N번** 발생하고, 네트워크 지연의 영향이 N배로 커진다. 파이프라이닝은 이 **RTT 횟수를 줄이는** 기법이다.

---

## 일반 실행 vs 파이프라이닝

### 일반 실행 (요청-응답을 반복)

- **흐름:** 명령 1 전송 → **응답 1 대기** → 명령 2 전송 → **응답 2 대기** → … 순서로, **한 명령마다 한 번의 왕복**이 일어난다.
- **특징:** 각 명령이 완료된 뒤에만 다음 명령을 보낼 수 있으므로, 명령이 N개면 **최소 N번의 RTT**가 필요하다. 네트워크 지연이 크면 전체 지연이 크게 늘어난다.

### 파이프라이닝 (일괄 전송·일괄 수신)

- **흐름:** **명령 1, 2, …, N을 응답을 기다리지 않고 연속으로 전송**한다. 서버는 도착한 순서대로 명령을 실행하고, **모든 실행이 끝난 뒤** (또는 버퍼 단위로) **응답 1, 2, …, N을 일괄로** 클라이언트에 보낸다. 클라이언트는 **응답 1, 2, …, N을 일괄로 수신**한다.
- **특징:** 여러 명령이 **한 번 또는 소수의 왕복**으로 처리되므로 **RTT가 크게 감소**한다. 동일한 N개 명령을 보낼 때 일반 실행보다 훨씬 짧은 시간에 끝낼 수 있다.

**정리:** 일반 실행은 “명령 1 → 응답 1 → 명령 2 → 응답 2”이고, 파이프라이닝은 “명령 1, 2, N 일괄 전송 → 응답 1, 2, N 일괄 수신”이다.

---

## 파이프라이닝 동작 순서

1. **클라이언트:** 명령 1, 2, 3(또는 N개)을 **한꺼번에**(또는 짧은 간격으로) 네트워크로 보낸다. 각 명령의 응답을 기다리지 않는다.
2. **네트워크:** 보낸 데이터가 Redis 서버로 전달된다.
3. **Redis 서버:** 도착한 명령을 **순서대로** 실행한다. 명령 1 실행 → 명령 2 실행 → 명령 3 실행. 실행 순서는 클라이언트가 보낸 순서와 같다.
4. **Redis 서버:** 실행이 끝난 명령들의 **응답을 순서대로** 클라이언트로 보낸다. (응답 1, 2, 3을 일괄 전송)
5. **네트워크:** 응답이 클라이언트에 전달된다.
6. **클라이언트:** 응답 1, 2, 3을 **일괄 수신**하고, 순서에 맞게 각 명령의 결과로 매핑한다.

전체 과정이 **한 번 또는 소수의 왕복**으로 이루어지므로, RTT로 인한 지연이 줄어든다. Redis는 별도의 “PIPELINE” 명령 없이, **클라이언트가 소켓에 명령을 연속으로 쓰고 응답을 한꺼번에 읽는** 방식으로 파이프라이닝을 구현한다. 대부분의 클라이언트 라이브러리는 `pipeline()` 또는 유사 API로 이 동작을 지원한다.

## 명령 일괄 처리: 파이프라이닝 vs 트랜잭션

여러 명령을 묶어서 보내는 방법에는 **파이프라이닝**과 **트랜잭션(MULTI/EXEC)**이 있다. 목적과 보장이 다르다.

| 구분 | 파이프라이닝 | 트랜잭션 (MULTI/EXEC) |
|------|--------------|------------------------|
| **목적** | **네트워크 최적화**, RTT 감소 | **원자성·격리** (한 덩어리로 실행, 다른 클라이언트 끼어들기 방지) |
| **실행 시점** | 명령이 서버에 **도착하는 대로 순차 실행**된다. | MULTI 이후 명령은 **큐에만 쌓이고**, **EXEC 시점**에 한꺼번에 실행된다. |
| **끼어들기** | 서버 입장에서는 “명령이 연속으로 들어오는” 것뿐이므로, **다른 클라이언트의 명령이 그 사이에 끼어들 수 있다.** | EXEC로 큐를 실행하는 동안 **다른 클라이언트의 명령은 끼어들 수 없다.** (격리) |
| **명령 관계** | 보통 **서로 독립적인 명령**을 많이 보낼 때 유리하다. (GET 여러 개, SET 여러 개 등) | **논리적으로 연관된 명령**을 “한 단위”로 실행하고 싶을 때 사용한다. (예: 읽기 → 계산 → 쓰기) |
| **원자성** | 보장하지 않는다. 명령 1이 실행된 뒤 다른 클라이언트 명령이 실행되고, 그 다음 명령 2가 실행될 수 있다. | MULTI~EXEC 블록은 **한 덩어리**로 실행되며, 그 안에서는 다른 클라이언트가 끼어들지 않는다. (단, Redis는 롤백 없음) |

- **파이프라이닝:** **RTT 감소**, 명령 **즉시 실행**, **끼어들기 가능**, **독립적 명령**에 적합.
- **트랜잭션:** **원자성·격리**, **EXEC 시점 실행**, **끼어들기 불가**, **연관된 명령**에 적합.

동시에 쓸 수 있다. MULTI, 명령들, EXEC를 **파이프라인에 넣어서** 한 번에 보내면, RTT를 줄이면서 트랜잭션의 격리 효과를 얻을 수 있다.

## 사용 시 유의사항

- **응답 순서:** 파이프라인으로 보낸 명령의 **응답 순서**는 **보낸 명령 순서**와 같다. 클라이언트는 “첫 번째 응답 = 첫 번째 명령의 결과”처럼 순서로 매핑하면 된다.
- **버퍼 크기:** 한 번에 너무 많은 명령을 파이프라인에 넣으면 클라이언트·서버 메모리와 네트워크 버퍼 부담이 커진다. 수백~수천 개 단위로 나누어 보내는 것이 안전하다.
- **에러 처리:** 파이프라인 중간에 잘못된 명령이 있으면, 해당 명령에서 오류가 반환된다. 클라이언트는 응답 배열을 순서대로 돌면서 각 결과가 정상인지 확인해야 한다.
- **트랜잭션과 병행:** “MULTI + 명령들 + EXEC”를 파이프라인에 넣어 한 번에 전송하면, RTT는 한 번이 되고 트랜잭션의 격리·일괄 실행은 그대로 유지된다.
