# Redis 키(Key) 조회 및 관리

Redis의 모든 데이터는 **키(key)**로 식별된다. 키 존재 여부 확인, 패턴으로 키 목록 조회, 만료 시간 설정 등 키 단위 연산을 정리한다.

---

## EXISTS (키 존재 확인)

- **동작:** `EXISTS key`는 지정한 **키가 현재 DB에 있는지** 확인한다.
- **반환값:**
  - 키가 **존재하면** **1**을 반환한다.
  - 키가 **없으면** **0**을 반환한다.
- **다중 키:** `EXISTS key1 key2 ...`처럼 여러 키를 주면, **존재하는 키의 개수**를 반환한다. 최대 2^32-1개까지 인자로 줄 수 있다.
- **시간 복잡도:** **O(1)**. 키 하나당 상수 시간이므로, 키 개수와 관계없이 빠르다.

캐시 히트 여부 판단, 키 삭제 전 존재 확인, 여러 키 중 몇 개가 있는지 셀 때 사용할 수 있다.

---

## 키 조회 명령어 (KEYS vs SCAN)

키 이름을 **패턴**으로 찾을 때 쓸 수 있는 명령은 **KEYS**와 **SCAN** 두 가지이다. 동작 방식과 프로덕션 사용 가능 여부가 다르다.

### KEYS (패턴 매칭, 전체 조회)

- **동작:** `KEYS pattern`은 **패턴에 맞는 모든 키**를 **한 번에** 반환한다.
- **패턴 매칭:** `*`(임의 문자 여러 개), `?`(문자 하나), `[abc]`(괄호 안 문자 중 하나) 등 글로브 스타일 패턴을 쓸 수 있다. 예: `KEYS user:*`, `KEYS session:???`.
- **전체 키 한 번에 조회:** 조건에 맞는 키를 **전부** 찾은 뒤 한 번에 클라이언트로 보낸다.
- **O(N) 및 서버 블로킹:** N은 **DB에 있는 전체 키 개수**이다. Redis는 단일 스레드이므로, KEYS가 실행되는 동안 **다른 명령을 처리하지 못한다**. 키가 수백만 개면 수 초 이상 블로킹될 수 있어 **프로덕션에서는 사용을 피하는 것이 좋다**.
- **활용:** 개발·디버깅, 키 개수가 적은 환경에서만 제한적으로 사용한다.

### SCAN (커서 기반, 반복 조회)

- **동작:** `SCAN cursor [MATCH pattern] [COUNT hint] [TYPE type]`은 **커서**를 이용해 키를 **조금씩** 나누어 조회한다. 한 번에 전체를 반환하지 않는다.
- **커서 기반:** 첫 호출은 보통 `SCAN 0`으로 시작한다. 반환값에 다음 커서가 오며, 0이 나올 때까지 같은 명령을 반복하면 전체를 순회한 것이 된다. 각 호출은 **짧은 시간**만 서버를 점유한다.
- **일부씩 나눠서 조회:** 한 번의 SCAN 호출마다 **일부 키만** 반환하므로, 대량의 키가 있어도 한 번에 긴 블로킹이 발생하지 않는다.
- **O(1) per call, 전체는 O(N):** 호출 한 번당 Redis 입장에서는 **상수 시간에 가깝게** 동작한다. 전체 키를 다 보려면 여러 번 호출해야 하므로 전체로 보면 O(N)이지만, **한 번에 N개를 모아서 처리하지 않아** 서버가 블로킹되지 않는다.
- **프로덕션에서 안전:** 키가 많아도 다른 요청에 미치는 영향이 적어 **운영 환경에서 키를 순회할 때 SCAN을 사용하는 것이 권장**된다.
- **MATCH:** KEYS와 비슷한 패턴을 쓸 수 있다. 단, 패턴 필터는 **반환할 키에만** 적용되므로, 실제로는 더 많은 키를 스캔한 뒤 그중 일부만 반환할 수 있다.
- **COUNT:** 한 번에 대략 몇 개씩 돌려줄지 힌트를 준다. 정확한 개수 보장은 아니다.

**정리:** 키 목록이 필요할 때는 **KEYS 대신 SCAN**을 사용하는 것이 안전하다.

## 기타 키 관련 명령

### DEL (키 삭제)

- `DEL key [key ...]`: 지정한 키를 **삭제**한다. 여러 키를 한 번에 줄 수 있다. **실제로 삭제된 키 개수**를 반환한다. 존재하지 않는 키는 무시된다.
- **시간 복잡도:** O(N). N은 삭제되는 키에 연결된 값의 크기 등에 따라 달라진다.

### TYPE (키의 타입)

- `TYPE key`: 해당 키에 저장된 **값의 타입**을 반환한다. `string`, `list`, `set`, `zset`, `hash`, `stream` 등 중 하나이다. 키가 없으면 `none`을 반환한다.
- **시간 복잡도:** O(1).

### TTL / PTTL (남은 만료 시간)

- `TTL key`: 키에 설정된 만료 시간의 **남은 초(second)**를 반환한다. 만료가 없으면 -1, 키가 없으면 -2를 반환한다.
- `PTTL key`: 남은 시간을 **밀리초** 단위로 반환한다.
- **시간 복잡도:** O(1).

### EXPIRE / PEXPIRE (만료 시간 설정)

- `EXPIRE key seconds`: 해당 키를 **지정한 초(seconds) 후** 만료되도록 설정한다.
- `PEXPIRE key milliseconds`: 밀리초 단위로 만료 시간을 설정한다.
- **시간 복잡도:** O(1).

### PERSIST (만료 제거)

- `PERSIST key`: 키에 설정된 **만료를 제거**하여 영구 키로 만든다. 만료가 없었으면 0, 제거에 성공하면 1을 반환한다.

## 사용 시 유의사항

- **KEYS 사용 자제:** 키 수가 많은 DB에서 KEYS를 호출하면 서버가 오래 블로킹되어 다른 요청이 지연된다. 키를 찾을 때는 **SCAN**을 사용한다.
- **SCAN의 일관성:** SCAN 도중 키가 추가·삭제되면 같은 키가 두 번 나오거나, 일부 키가 한 번의 순회에서 빠질 수 있다. “전체 스냅샷”이 아니라 “대략적인 순회”로 본다.
- **EXISTS와 삭제:** 키가 있는지 확인한 뒤 DEL하는 것은 별도 명령이므로, 그 사이에 다른 클라이언트가 키를 바꿀 수 있다. “있을 때만 지우기” 등이 필요하면 Lua 스크립트나 트랜잭션을 고려한다.
