# Redis 영속성 (Persistence)

Redis는 인메모리 저장소이므로 서버가 종료되면 메모리 위의 데이터는 사라진다. **영속성(persistence)** 설정을 통해 디스크에 데이터를 남겨 두고, 재시작 시 복구할 수 있다. RDB, AOF, Hybrid 방식을 정리한다.

---

## 영속성과 데이터 복구 흐름

Redis 서버는 **메모리에 데이터를 적재**한 뒤 클라이언트 요청을 처리한다. 서버가 **종료되거나 재시작**되면 메모리 내용은 초기화된다.

- **영속성 설정이 없음:** 디스크에 저장된 스냅샷이나 로그가 없으면, 재시작 후 **이전 데이터를 복구할 수 없다**. 메모리에 있던 데이터는 **손실**된다.
- **영속성 설정이 있음:** RDB 파일이나 AOF 파일이 디스크에 있으면, 재시작 시 Redis가 **디스크에서 데이터를 읽어 복구**한 뒤 다시 **메모리에 적재**한다. 이전 상태로 서비스를 이어갈 수 있다.

따라서 재시작 후에도 데이터를 유지하려면 **RDB**, **AOF**, 또는 **Hybrid** 중 하나 이상을 설정해야 한다.

---

## RDB 스냅샷 (Snapshot)

RDB는 **특정 시점의 메모리 데이터 전체**를 바이너리 파일(기본값 `dump.rdb`)로 저장하는 방식이다. 스냅샷 한 번으로 “그 순간의 DB”를 디스크에 남긴다.

### SAVE (동기 저장)

- **동작:** `SAVE` 명령을 실행하면 Redis가 **즉시** 현재 메모리 상태를 RDB 파일로 기록한다. 이 작업이 **끝날 때까지** 서버는 다른 명령을 처리하지 않는다.
- **서버 차단:** 저장이 진행되는 동안 **모든 클라이언트 요청이 블로킹**된다. 데이터가 많을수록 차단 시간이 길어진다.
- **운영 환경 비추천:** 프로덕션에서는 가용성과 응답 시간을 위해 **SAVE 사용을 피하는 것이 좋다**. 긴급 백업 등 특수한 경우에만 제한적으로 사용한다.

### BGSAVE (백그라운드 저장)

- **동작:** `BGSAVE`를 실행하면 Redis가 **자식 프로세스를 하나 만든 뒤**, 그 자식이 RDB 파일을 디스크에 쓴다. 부모(메인) 프로세스는 **그동안 클라이언트 요청을 계속 처리**한다.
- **fork 사용:** 자식 프로세스 생성 시 **fork()**를 사용한다. fork 시점의 메모리 페이지를 자식이 공유하고, 자식만 그 내용을 파일로 기록한다.
- **Copy-on-Write (CoW):** fork 이후 부모가 메모리 페이지를 수정하면, 해당 페이지만 복사되어 부모가 사용하고, 자식은 수정 전 데이터를 그대로 디스크에 쓴다. 이를 통해 메모리 사용과 블로킹을 줄인다.
- **운영 환경 권장:** 스냅샷 방식으로 백업할 때는 **BGSAVE**를 사용한다. 수동으로 호출하거나, 아래 자동 저장 조건으로 간접적으로 호출된다.

### 자동 저장 (save 설정)

- **동작:** `redis.conf`의 **save** 지시어로 “N초 동안 M번 이상 키가 변경되면 자동으로 RDB 저장”을 설정할 수 있다. 예: `save 900 1`(15분 동안 1번 이상 변경 시), `save 300 10`, `save 60 10000`.
- **조건 만족 시:** 설정된 조건이 충족되면 Redis가 내부적으로 **BGSAVE**를 실행한다. 즉, 자동 저장도 **백그라운드**로 동작하여 서버를 블로킹하지 않는다.
- **의미:** 주기적으로 또는 변경이 많을 때 자동으로 스냅샷이 생성되어, 장애 시 “최근 스냅샷”까지는 복구할 수 있다. 다만 스냅샷 간 구간의 데이터는 손실될 수 있다.

**정리:** RDB는 **한 번에 전체 스냅샷**을 남기므로 복구가 빠르고 파일이 단순하지만, 스냅샷 주기 사이에 발생한 변경은 잃을 수 있다.

## AOF 로그 (Append Only File)

AOF는 **쓰기 명령(및 그 인자)**을 **추가 전용(append-only)** 로그 파일에 기록하는 방식이다. 재시작 시 이 로그를 순서대로 재실행하여 상태를 복구한다. RDB보다 손실 구간을 줄일 수 있다.

### appendfsync 정책

AOF에 쓴 내용을 **언제 디스크에 반영(동기화)할지**는 **appendfsync**로 결정한다.

| 정책 | 동작 | 특성 |
|------|------|------|
| **always** | **명령마다** fsync를 호출하여 디스크에 동기화한다. | **최대 안전성.** 크래시 시에도 마지막 명령까지 유지될 가능성이 높다. 대신 쓰기 부하가 크고 성능이 떨어진다. |
| **everysec** | **1초마다** 한 번씩 fsync를 수행한다. | 안전성과 성능의 균형. **일반적으로 권장**하는 설정이다. 최대 1초 분량의 쓰기는 손실될 수 있다. |
| **no** | Redis는 버퍼만 채우고, **실제 디스크 동기화 시점은 OS에 맡긴다.** | **최대 성능.** 동기화가 지연될 수 있어 크래시 시 손실 가능 구간이 더 길어질 수 있다. |

### BGREWRITEAOF (AOF 재작성)

- **동작:** AOF는 “명령을 계속 추가”하는 형태이므로, 오래 실행될수록 파일이 커지고 같은 키에 대한 중복 명령이 많아진다. **BGREWRITEAOF**는 **현재 메모리 상태를 반영한 최소한의 명령**으로 AOF를 **다시 작성**한다. 백그라운드에서 수행되며, 완료 후 새 AOF가 기존을 대체한다.
- **파일 압축:** 효과적으로 “AOF 파일 압축”이라 볼 수 있다. 자동 재작성 조건은 `redis.conf`의 `auto-aof-rewrite-percentage`, `auto-aof-rewrite-min-size` 등으로 설정한다.

**정리:** AOF는 **쓰기 명령 로그**이므로 RDB보다 상대적으로 **데이터 손실을 줄일 수 있지만**, 복구 시 로그를 처음부터 재실행해야 하므로 데이터가 많으면 **복구 시간이 길어질** 수 있다.

## Hybrid 방식 (RDB + AOF)

Redis 4.0부터 **AOF 재작성 시** Hybrid 방식을 사용할 수 있다. 재작성된 AOF 파일의 **앞부분은 RDB 스냅샷**, **뒷부분은 AOF 형식의 증분 로그**로 구성된다.

- **앞부분: RDB 스냅샷**  
  재작성 시점의 전체 스냅샷을 RDB와 동일한 바이너리 형식으로 저장한다. 이 부분 덕분에 **복구 시 스냅샷을 먼저 로딩**할 수 있어 **로딩 속도가 빨라진다**.

- **뒷부분: AOF 로그**  
  재작성 이후 발생한 쓰기 명령만 AOF 형식으로 이어 붙인다. 스냅샷 이후 ~ 크래시 직전까지의 **데이터 손실을 최소화**한다.

- **결과:** **빠른 로딩**(RDB 이점)과 **최소 데이터 손실**(AOF 이점)을 동시에 얻어, **복구 성능과 안전성**을 함께 높인다. AOF를 사용하면서 재작성이 켜져 있으면, 재작성된 파일은 이 Hybrid 형식으로 생성된다.

**정리:** AOF + 재작성을 사용하는 경우, 실제 디스크에 남는 파일은 “RDB 한 블록 + AOF 꼬리” 형태가 되며, 이 구성을 Hybrid 방식이라 부른다.

---

## 방식 비교 및 선택

| 구분 | RDB | AOF | Hybrid |
|------|-----|-----|--------|
| **저장 단위** | 시점 스냅샷 | 쓰기 명령 로그 | 재작성 시 RDB + AOF 꼬리 |
| **복구 속도** | 빠름 (한 번 로딩) | 느릴 수 있음 (로그 재실행) | 빠름 (스냅샷 먼저 로딩) |
| **손실 가능 구간** | 마지막 스냅샷 ~ 크래시 | appendfsync에 따라 수 초 이내 등 | 최소화 |
| **파일 크기** | 보통 작음 | 커질 수 있음 (재작성으로 축소) | 재작성 후 보통 작음 |
| **운영 권장** | BGSAVE/자동 save 사용, SAVE 비권장 | appendfsync everysec, 재작성 활성화 | AOF + 재작성 사용 시 자동 적용 |

재시작 후 데이터를 복구해야 하면 **최소한 RDB 또는 AOF 중 하나**를 켜 두고, 가능하면 **AOF(everysec) + 재작성(Hybrid)** 구성을 권장한다.
