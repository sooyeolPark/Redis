# Redis String 타입

Redis의 String 타입은 키-값 저장의 기본이 되는 데이터 타입이다. 이름과 달리 텍스트뿐 아니라 숫자, 이진 데이터, 직렬화된 객체까지 저장할 수 있다.

---

## String 타입의 주요 특징

### 1. 가장 기본적인 데이터 타입

Redis는 키-값 스토어이며, **모든 키는 String 타입**이다. 값 역시 많은 경우 String 타입으로 저장된다.

- **키-값 저장:** 키로 값을 저장하고 조회하는 가장 단순한 모델을 따른다.
- **단순하고 직관적:** SET/GET만으로 저장·조회가 가능하여 초보자도 쉽게 사용할 수 있다. 캐시, 세션, 설정값, 단순 카운터 등 다양한 용도에 맞춰 활용할 수 있다.

### 2. 바이너리 세이프 (Binary Safe)

String 타입은 **바이너리 세이프**이므로, 특정 문자 인코딩을 강제하지 않고 임의의 바이트 시퀀스를 그대로 저장·조회할 수 있다.

- **텍스트:** UTF-8 등 다양한 인코딩의 문자열을 손실 없이 저장한다. 사용자 이름, 이메일, HTML 조각 등에 적합하다.
- **숫자:** 숫자를 문자열 형태로 저장할 수 있다. Redis는 필요 시 이를 숫자로 해석하여 INCR/DECR 같은 연산을 수행한다. 카운터나 ID 생성에 유용하다.
- **이진 데이터:** 이미지, 오디오, 동영상 등 이진 데이터를 그대로 저장할 수 있다. 경량 캐시나 임시 스토리지로 쓸 때 활용된다.
- **직렬화된 객체:** JSON, MessagePack, Protocol Buffers 등으로 직렬화한 객체를 String으로 저장하고, 애플리케이션에서 역직렬화하여 사용할 수 있다.

### 3. 최대 512MB 크기 제한

한 키에 저장할 수 있는 String 값의 크기는 **최대 512MB**이다.

- **상한선:** 단일 값이 512MB를 넘을 수 없으며, 그 이상은 다른 방식(스트림, 파일 스토리지 등)을 고려하여야 한다.
- **작은 값 권장:** Redis는 인메모리 저장소이므로, 성능과 메모리 효율을 위해 가능하면 작은 값을 쓰는 것이 좋다. 값이 크면 메모리와 네트워크 전송 부담이 커진다.
- **메모리 효율:** 작은 값을 사용하면 메모리 단편화를 줄이고 전체 메모리 사용 효율을 높일 수 있다.

## SET 명령어

SET은 지정한 **키에 값을 저장**하는 기본 명령어이다.

- **키에 값 저장:** 키가 없으면 **새 키를 만들고** 값을 넣는다.
- **기존 값 덮어쓰기:** 키가 이미 있으면 기존 값을 **새 값으로 덮어쓴다**. 별도의 “업데이트” 명령 없이 SET만으로 갱신할 수 있다.
- **항상 성공:** 메모리 부족 등 예외 상황이 없다면 SET은 항상 성공한다. 키 존재 여부와 관계없이 값을 설정한다.

---

## GET 명령어

GET은 지정한 **키에 저장된 값을 조회**하는 기본 명령어이다.

- **키의 값 조회:** 키를 인자로 주면 해당 키에 저장된 값을 반환한다.
- **존재하지 않으면 nil 반환:** 키가 없으면 에러가 아니라 **nil**을 반환한다. 이를 통해 “키 없음”을 구분할 수 있다.
- **String 타입만 조회 가능:** GET은 **String 타입**인 키에만 사용할 수 있다. List, Hash, Set 등 다른 타입의 키에 GET을 쓰면 타입 오류가 발생하거나 예상과 다른 결과가 나올 수 있다.

## SET 명령어 옵션

SET은 단순 저장 외에 **만료 시간(TTL)**과 **조건부 저장(NX/XX)** 옵션을 함께 쓸 수 있다.

### SETEX (SET with Expire)

키에 값을 넣으면서 **만료 시간을 초 단위로 설정**한다.

- **TTL 함께 설정:** 키를 저장할 때 동시에 TTL(Time-To-Live)을 부여한다. TTL이 지나면 Redis가 해당 키를 **자동으로 삭제**한다.
- **자동 만료:** 애플리케이션에서 따로 삭제하지 않아도 되므로, 일정 시간만 유효한 데이터를 다루기 편하다.
- **활용:** **캐싱**에 적합하다. 일정 시간 동안만 유효한 응답 캐시, 세션 데이터 등을 저장할 때 사용한다.

### SETNX (SET if Not eXists)

**키가 없을 때만** 값을 저장한다.

- **키가 없을 때만 저장:** 키가 이미 존재하면 SET을 수행하지 않고, 없을 때만 값을 설정한다. 중복 생성이 필요 없는 키를 안전하게 만들 수 있다.
- **중복 방지:** 같은 키를 여러 클라이언트가 동시에 만들려 할 때, 한 번만 성공하고 나머지는 실패하도록 할 수 있다.
- **활용:** **분산 락(Distributed Lock)** 구현에 핵심적으로 쓰인다. 락 키를 SETNX로 한 번만 설정하고, 성공한 클라이언트만 락을 획득한 것으로 본다.

### SET 옵션 (EX, PX, NX, XX)

Redis 2.6.12 이후 SET 하나로 위 기능을 조합할 수 있다.

| 옵션 | 설명 |
|------|------|
| **EX \<초\>** | 만료 시간을 **초** 단위로 설정한다. SETEX와 동일한 TTL 기능을 제공한다. |
| **PX \<밀리초\>** | 만료 시간을 **밀리초** 단위로 설정한다. 더 짧은 TTL이 필요할 때 사용한다. |
| **NX** | **키가 없을 때만** 저장한다. SETNX와 동일하다. |
| **XX** | **키가 이미 있을 때만** 값을 덮어쓴다. 키가 없으면 아무 동작도 하지 않는다. 기존 키만 업데이트할 때 사용한다. |

예: `SET key value EX 300 NX` — 키가 없을 때만 저장하고, 300초 후 만료한다.

## 카운터 명령어 (INCR, DECR, INCRBY, DECRBY)

숫자 형태의 String 값에 대해 **증감 연산**을 수행하는 명령어이다. 모두 **원자적(atomic)**으로 동작한다.

| 명령어 | 동작 | 활용 예 |
|--------|------|---------|
| **INCR** | 값을 **1 증가**시킨다. | 조회수, 좋아요 수, 접속 수 |
| **DECR** | 값을 **1 감소**시킨다. | 재고 감소, 남은 수량 |
| **INCRBY** | 값을 **지정한 정수 N만큼** 증가시킨다. | 포인트 적립, 일괄 증가 |
| **DECRBY** | 값을 **지정한 정수 N만큼** 감소시킨다. | 다운로드 잔여 횟수, 쿼터 차감 |

- **원자적 연산:** 한 번의 명령이 나눠지지 않고 한 번에 실행된다. 여러 클라이언트가 동시에 INCR을 호출해도 값이 정확히 증가하며, **레이스 컨디션**을 피할 수 있다.
- **동시성 보장:** 락 없이도 카운터의 일관성을 유지할 수 있어, 분산 환경에서 조회수·재고·포인트 등을 안전하게 다룰 수 있다.
- **주의:** 해당 키의 값이 정수로 해석 가능한 문자열이어야 한다. 그렇지 않으면 오류가 발생한다.

## 다중 작업 명령어 (MSET, MGET)

여러 키에 대한 쓰기나 읽기를 **한 번의 명령**으로 처리하는 명령어이다.

### MSET (Multiple SET)

**여러 키-값 쌍을 한 번에 저장**한다.

- 여러 개의 `키 값` 쌍을 인자로 넘기면, 한 번의 MSET 호출로 모두 저장된다.
- **원자적 실행:** MSET에 포함된 모든 키-값 설정이 **한 번에** 적용된다. 중간에 실패하면 전체가 적용되지 않아 데이터 일관성을 유지할 수 있다.

### MGET (Multiple GET)

**여러 키의 값을 한 번에 조회**한다.

- 조회할 키 목록을 인자로 주면, 각 키에 대한 값이 **순서대로** 배열로 반환된다. 키가 없으면 해당 위치에 nil이 온다.

### 공통 효과: 네트워크 RTT 감소

- SET/GET을 반복하면 요청마다 **네트워크 왕복(RTT)**이 발생한다. MSET/MGET을 쓰면 여러 작업을 **한 번의 왕복**으로 처리할 수 있다.
- RTT가 줄어들어 **성능이 향상**되고, 클라이언트·서버 모두 **효율적으로 처리**할 수 있다. 캐시 워밍, 프로필/설정 일괄 조회 등에 적합하다.

## 내부 구조: SDS와 인코딩

Redis는 문자열을 C의 null 종료 문자열이 아니라 **SDS(Simple Dynamic String)**라는 자체 구조로 저장한다.

### SDS 구조

- **len (int):** 현재 문자열의 길이. 길이 조회가 **O(1)**이다.
- **free (int):** 버퍼 내 사용 가능한 여유 공간. APPEND 시 재할당을 줄이고 **버퍼 오버플로우**를 방지하는 데 쓰인다.
- **buf (char[]):** 실제 바이트 배열.

**특징:** O(1) 길이 조회, 버퍼 오버플로우 방지, **바이너리 세이프**(널 문자를 포함한 임의의 바이트 저장 가능), APPEND 연산 시 여유 공간을 활용한 효율적인 확장.

### 세 가지 인코딩 방식

Redis는 값의 내용과 길이에 따라 메모리 효율을 위해 서로 다른 인코딩을 사용한다.

| 인코딩 | 대상 | 설명 |
|--------|------|------|
| **int** | 64비트 부호 있는 정수 범위의 숫자 문자열 | 객체 헤더와 정수만 저장하여 메모리를 절약한다. 예: `SET key 123` |
| **embstr** | 길이 **44바이트 이하**인 문자열 | Redis 객체와 문자열 데이터를 **한 메모리 블록**에 연속 저장한다. 캐시 친화적이며, 생성 후에는 읽기 전용으로 취급된다. |
| **raw** | 44바이트를 **초과**하는 문자열, 또는 수정이 일어난 embstr | 객체와 문자열이 **분리된** 메모리에 있고, 수정·APPEND가 가능하다. |

### 인코딩 자동 전환

- **int → raw:** 정수로 저장된 값에 `APPEND`로 숫자가 아닌 문자열을 붙이거나, 정수 범위를 벗어나는 문자열이 되면 **raw**로 바뀐다. 예: `SET key 123` 후 `APPEND key 456` → `'123456'` (raw).
- **embstr → raw:** embstr는 수정이 불가능하므로, **APPEND** 등으로 값을 바꾸면 내부적으로 **raw**로 전환한 뒤 수정한다. 예: `SET key 'hello'` 후 `APPEND key 'world'` → `'helloworld'` (raw).

이를 통해 Redis는 문자열의 길이와 사용 패턴에 맞게 메모리를 절약하면서도, 필요 시 유연하게 수정·확장할 수 있다.
