# Redis Pub/Sub (Publish/Subscribe)

Redis Pub/Sub은 **채널(channel)**을 매개로 **발행자(publisher)**가 메시지를 보내고, **구독자(subscriber)**가 실시간으로 받는 메시징 패턴이다. 발행자와 구독자는 서로를 알 필요 없이 **채널 이름(또는 패턴)**만으로 연결된다. 알림, 이벤트 브로드캐스트, 실시간 대시보드 등에 활용할 수 있다.

---

## Pub/Sub 아키텍처

- **발행자(Publisher):** 특정 **채널**로 메시지를 **발행(PUBLISH)**한다. 누가 그 메시지를 받는지는 알지 못한다.
- **채널(Channel):** 메시지가 전달되는 **중간 매개체**이다. 발행자가 채널에 메시지를 보내면, 그 채널을 구독 중인 모든 구독자에게 **실시간으로 전달**된다.
- **구독자(Subscriber):** 원하는 **채널(또는 패턴)**을 **구독(SUBSCRIBE / PSUBSCRIBE)**하고, 해당 채널에 발행된 메시지를 **실시간으로 수신**한다. 누가 보냈는지는 알 필요 없다.
- **서로 모름(Decoupling):** 발행자는 구독자의 존재·개수를 알지 못하고, 구독자도 발행자를 특정하지 않는다. **채널**만 공유하면 되므로, 구성 요소를 독립적으로 추가·제거하기 쉽다.

이 구조를 **발행-구독 패턴**이라 부르며, Redis는 이를 위한 명령 SUBSCRIBE, PSUBSCRIBE, PUBLISH, UNSUBSCRIBE, PUNSUBSCRIBE를 제공한다.

---

## Pub/Sub 기능 및 명령

### 일반 구독 (SUBSCRIBE)

- **동작:** `SUBSCRIBE channel [channel ...]`은 **지정한 채널 이름**을 **정확히** 구독한다. 해당 채널에 PUBLISH된 메시지만 수신한다.
- **정확한 채널명:** 채널 이름이 완전히 일치해야 한다. 예: `SUBSCRIBE news`는 `news` 채널만 구독하며, `news:sports` 등은 별도 채널로 취급된다.
- **다중 채널:** 한 클라이언트가 여러 채널을 동시에 구독할 수 있다. 구독 모드에 들어가면 해당 연결에서는 SUBSCRIBE, UNSUBSCRIBE, PSUBSCRIBE, PUNSUBSCRIBE, PING, QUIT 등만 사용할 수 있고, 그 밖의 명령(SET, GET 등)은 사용할 수 없다.

### 패턴 구독 (PSUBSCRIBE)

- **동작:** `PSUBSCRIBE pattern [pattern ...]`은 **패턴**으로 채널을 구독한다. 패턴에 **와일드카드**를 써서 이름이 매칭되는 모든 채널의 메시지를 받을 수 있다.
- **와일드카드 매칭:** `*`는 임의의 문자 열에 매칭된다. 예: `PSUBSCRIBE news:*`는 `news:weather`, `news:sports`, `news:tech` 등 `news:`로 시작하는 모든 채널의 메시지를 수신한다. `?`는 문자 하나에 매칭된다.
- **활용:** 채널 이름에 네임스페이스(예: `news:weather`, `news:sports`)를 두고, 한 번에 `news:*`만 구독하면 해당 계열 채널을 모두 받을 수 있다.

### 메시지 발행 (PUBLISH)

- **동작:** `PUBLISH channel message`는 지정한 **채널**에 **메시지**를 발행한다. 그 순간 해당 채널을 구독 중인 **모든 클라이언트**에게 메시지가 **실시간으로 전달**된다.
- **반환값:** 메시지를 **수신한 구독자 수**를 반환한다. 구독자가 없으면 0이다.
- **실시간 전달:** Redis는 발행된 메시지를 큐에 쌓아 두지 않고, 구독 중인 연결에 즉시 전달한다. 따라서 구독자가 그 순간 연결되어 있어야 수신할 수 있다.

### 구독 해제 (UNSUBSCRIBE, PUNSUBSCRIBE)

- **UNSUBSCRIBE [channel ...]:** 지정한 채널 구독을 해제한다. 채널을 생략하면 모든 채널 구독을 해제한다.
- **PUNSUBSCRIBE [pattern ...]:** 지정한 패턴 구독을 해제한다. 패턴을 생략하면 모든 패턴 구독을 해제한다.

## Redis 서버 내부 구조

Redis는 Pub/Sub 구독 정보를 두 가지 자료구조로 관리한다.

- **pubsub_channels (Dictionary):** **채널 이름 → 구독자 리스트**를 매핑한다. `SUBSCRIBE news`를 한 클라이언트는 `news` 채널의 구독자 리스트에 추가된다. `PUBLISH news "hello"` 시 이 리스트에 있는 모든 클라이언트에게 메시지를 보낸다. 예: `news` → 구독자 리스트, `sports` → 구독자 리스트.
- **pubsub_patterns (List):** **패턴 → 구독자** 정보를 리스트로 가진다. `PSUBSCRIBE news*`를 한 클라이언트는 이 리스트에 등록된다. 메시지 발행 시 채널 이름이 패턴에 매칭되는지 확인하고, 매칭되는 패턴의 구독자들에게도 메시지를 전달한다. 예: 패턴 `news*` → 구독자 리스트.

이렇게 **정확한 채널 구독**과 **패턴 구독**을 분리해 저장하고, PUBLISH 시 두 구조를 모두 참고해 수신 대상을 결정한다.

## 클러스터 환경에서의 Pub/Sub

Redis Cluster에서 Pub/Sub 메시지는 **클러스터 버스(Cluster Bus)**를 통해 전파될 수 있다.

- **발행자:** 한 노드(예: Node A)에서 `PUBLISH channel message`를 실행하면, 해당 메시지는 **클러스터 버스**를 타고 다른 노드로 전달된다.
- **클러스터 버스:** 노드 간 내부 통신·상태 동기화에 쓰이는 채널이다. Pub/Sub 메시지도 이 경로로 **다른 노드들(Node A, B, C 등)**에 전달된다.
- **구독자 배포:** 각 노드는 자신에게 연결된 **구독자**에게만 메시지를 전달한다. Node A에 연결된 구독자 1, Node B의 구독자 2, Node C의 구독자 3처럼 노드가 나뉘어 있어도, 한 노드에서 발행한 메시지가 클러스터 버스를 통해 모든 노드로 퍼지고, 각 노드가 자기 구독자에게 전달한다.

따라서 클러스터에서도 **어느 노드에 연결했든** 같은 채널을 구독하면 발행된 메시지를 받을 수 있다. (구체적인 동작은 Redis 버전과 설정에 따라 다를 수 있다.)

## 메시지 전달 보장 수준 (At-Most-Once)

Redis Pub/Sub은 **메시지를 저장하지 않고** “그 순간 연결된 구독자”에게만 전달한다. 따라서 전달 보장 수준은 **At-Most-Once**(최대 1번)에 해당한다.

- **의미:** 각 메시지는 **0번** 전달되거나 **1번** 전달된다. 구독자가 없으면 0번, 있으면 그때 연결된 구독자에게 1번 전달된다. **재전송·재시도·영속화가 없으므로** “최소 1번”이나 “정확히 1번”을 보장하지 않는다.
- **구독자가 없을 때:** PUBLISH 시 수신자 수는 0이다. 나중에 구독자가 붙어도 **과거 메시지는 받을 수 없다**.
- **구독자 연결 끊김:** 메시지가 전달되기 직전에 연결이 끊기면 그 메시지는 **손실**된다. 재연결 후에도 다시 받을 수 없다.
- **다른 시스템과 비교:**  
  - **At-Least-Once(최소 1번):** RabbitMQ ACK 등으로 메시지를 보관·재전송하여 최소 한 번은 전달을 시도한다. 중복 가능.  
  - **Exactly-Once(정확히 1번):** Kafka 트랜잭션 등으로 중복 없이 한 번만 처리되도록 보장한다.  
  Redis Pub/Sub은 이 둘보다 보장이 약하고, **실시간 브로드캐스트**에 적합하다.

**활용 시 유의:** 중요한 메시지는 Redis Pub/Sub만으로는 유실될 수 있으므로, 필요하면 메시지 큐(Redis Stream, RabbitMQ, Kafka 등)와 조합하거나, “있으면 좋은” 수준의 알림·이벤트에 사용하는 것이 안전하다.

---

## 사용 시 유의사항

- **구독 모드:** 클라이언트가 SUBSCRIBE 또는 PSUBSCRIBE를 실행하면 해당 연결은 **구독 전용 모드**로 전환된다. 이 연결로는 GET, SET 등 일반 명령을 보낼 수 없다. 발행은 **별도 연결**에서 PUBLISH로 수행한다. (예제처럼 subscriber용 클라이언트와 publisher용 클라이언트를 나누는 것이 일반적이다.)
- **영속성 없음:** Pub/Sub 메시지는 디스크에 저장되지 않으며, AOF/RDB에도 기록되지 않는다. 재시작 후 복구할 수 없다.
- **클라이언트 라이브러리:** ioredis 등에서는 `subscribe()`, `on('message', ...)`, `publish()` 형태로 제공한다. 패턴 구독은 `psubscribe()`, `on('pmessage', ...)`로 처리한다.
