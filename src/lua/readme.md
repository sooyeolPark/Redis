# Redis Lua 스크립팅

Redis는 **Lua** 스크립트를 서버에서 실행하는 기능을 제공한다. 스크립트는 **원자적으로** 실행되며, 조건문·반복문·계산을 써서 복잡한 로직을 한 번에 처리할 수 있고, **한 번의 호출**로 여러 Redis 명령을 실행하여 **RTT와 네트워크 부담**을 줄일 수 있다. EVAL로 스크립트를 넘겨 실행하고, EVALSHA로 캐시된 스크립트를 해시만으로 실행할 수 있다.

---

## Lua 스크립팅의 장점

### 1. 원자적 실행 — 트랜잭션 대체

- **원자적 실행:** Lua 스크립트는 **시작부터 끝까지** Redis 단일 스레드 위에서 **끊기지 않고** 실행된다. 스크립트가 도는 동안 **다른 클라이언트의 명령은 처리되지 않는다**.
- **중간 상태 없음:** “명령 1 실행 → (다른 클라이언트 끼어듦) → 명령 2 실행” 같은 상황이 없다. 스크립트 안의 모든 Redis 호출이 **한 덩어리**로 실행된다.
- **격리성 보장:** 트랜잭션(MULTI/EXEC)과 비슷하게 **격리**된 실행이 이루어지며, 조건문·반복·에러 처리까지 스크립트 안에서 제어할 수 있어 **트랜잭션 대체** 용도로 많이 쓴다. (Redis 트랜잭션은 롤백이 없지만, Lua에서는 로직으로 “실패 시 아무 것도 하지 않음”을 구현할 수 있다.)

### 2. 복잡한 로직 — 서버 측 실행

- **서버 측 실행:** 조건 분기, 반복, 산술 연산 등이 **Redis 서버**에서 수행된다. 클라이언트는 “스크립트 한 번”만 보내면 된다.
- **조건문·반복문:** Lua 문법의 `if`/`elseif`/`else`, `for`, `while`을 그대로 쓸 수 있다. “값을 읽고 조건에 따라 SET할지 말지” 같은 로직을 스크립트 안에 넣을 수 있다.
- **계산 가능:** `tonumber()`, `+`, `-`, `*`, `/` 등으로 **서버에서 계산**한 뒤 그 결과를 Redis 명령의 인자로 쓸 수 있다. 클라이언트에서 계산해 여러 번 왕복할 필요가 줄어든다.

### 3. 네트워크 절감 — RTT 감소

- **한 번의 호출:** 여러 Redis 명령을 **한 개의 EVAL(또는 EVALSHA) 호출**로 묶어 보낸다. 클라이언트–서버 왕복이 **한 번**이면 된다.
- **여러 명령 실행:** 스크립트 안에서 `redis.call('GET', ...)`, `redis.call('SET', ...)` 등을 여러 번 호출해도, 네트워크 상으로는 **한 번**만 오간다. RTT가 줄고 처리량이 좋아진다.

---

## EVAL 실행 과정

1. **EVAL 명령:** 클라이언트가 `EVAL script numkeys key [key ...] arg [arg ...]` 형태로 **스크립트 본문**과 키·인자를 보낸다.
2. **스크립트 전송:** 스크립트 문자열이 Redis 서버에 전달된다.
3. **Lua 인터프리터:** Redis 서버는 내장 **Lua 인터프리터**로 해당 스크립트를 파싱·실행한다.
4. **스크립트 실행 + 원자적 실행:** 스크립트가 실행되는 동안 Redis는 **다른 명령을 처리하지 않는다**. 즉, **원자적 실행**이며 **다른 명령이 끼어들 수 없다**.
5. **Redis 명령 호출:** 스크립트 안에서 `redis.call('GET', key)` 또는 `redis.pcall(...)`로 **Redis 명령**을 호출한다. 호출된 명령은 그 자리에서 순서대로 실행되고, 그 결과가 Lua 쪽으로 반환된다.
6. **결과 반환:** 스크립트가 `return ...`으로 반환한 값이 **EVAL의 최종 반환값**이 되어 클라이언트에게 전달된다.

정리하면, EVAL 한 번으로 “스크립트 전송 → Lua로 실행(원자) → Redis 명령들 호출 → 결과 반환”이 한 흐름으로 이루어진다.

## Lua 문법 요약

Redis에서 쓰는 Lua는 표준 Lua 5.1 문법을 따른다. 스크립트 작성 시 자주 쓰는 요소만 정리한다.

| 구분 | 설명 |
|------|------|
| **변수** | `local`로 지역 변수를 선언한다. 타입을 붙이지 않는 **동적 타입**이다. 예: `local x = 1`, `local key = KEYS[1]`. |
| **조건문** | `if ... then ... elseif ... then ... else ... end` 형태로 분기한다. |
| **반복문** | `for`, `while`을 쓸 수 있다. 예: `for i = 1, 10 do ... end`. |
| **테이블** | Lua의 **테이블**은 **배열**이자 **해시**이다. 숫자 인덱스는 배열처럼, 문자열 키는 맵처럼 쓴다. 예: `t[1]`, `t['name']`. |
| **함수** | `function name(arg) ... return ... end` 또는 `local f = function() ... end`. 함수는 **일급 객체**라 변수에 넣거나 인자·반환값으로 쓸 수 있다. |

Redis 스크립트에서는 **KEYS**, **ARGV** 테이블로 EVAL에 넘긴 키와 인자를 받고, **redis.call()** / **redis.pcall()**로 Redis 명령을 호출한다. `tonumber()`, `tostring()` 등으로 타입 변환을 한다.

## SCRIPT LOAD와 EVALSHA (스크립트 캐시)

매번 긴 스크립트 문자열을 EVAL로 보내면 네트워크와 파싱 비용이 반복된다. **스크립트를 서버에 캐시**하고 **해시만으로 실행**하려면 SCRIPT LOAD와 EVALSHA를 쓴다.

### SCRIPT LOAD

- **동작:** `SCRIPT LOAD script`는 Lua 스크립트 문자열을 Redis 서버에 **등록**하고, 그 스크립트의 **SHA1 해시**를 반환한다.
- **서버에 캐싱:** 로드된 스크립트는 서버 메모리에 **해시 → 스크립트** 형태로 캐시된다. 재시작 전까지 같은 해시로 다시 실행할 수 있다.
- **흐름:** 스크립트(문자열) → **SHA1 해시 계산** → SCRIPT LOAD로 서버에 전송 → **서버에 캐싱** → 해시 반환.

### EVALSHA

- **동작:** `EVALSHA sha1 numkeys key [key ...] arg [arg ...]`는 **이미 서버에 올라온 스크립트**를 **해시(sha1)**만 지정해서 실행한다. 스크립트 본문을 다시 보내지 않는다.
- **해시로 실행:** 인자로 넘긴 SHA1이 캐시에 있으면 해당 스크립트가 실행되고, 없으면 오류가 반환된다. (그때는 EVAL로 스크립트를 다시 보내거나, SCRIPT LOAD 후 EVALSHA를 쓰면 된다.)
- **캐시 활용:** 한 번 SCRIPT LOAD(또는 EVAL)로 캐시된 스크립트는 **EVALSHA + 해시**만으로 반복 실행할 수 있어, 네트워크 전송량과 파싱 비용이 줄어든다.

**정리:** 자주 쓰는 스크립트는 앱 기동 시 SCRIPT LOAD로 올려 두고, 이후에는 EVALSHA로만 호출하는 패턴이 **최적화 전략** 중 하나이다.

## 최적화 전략

### 1. 명령 최소화

- **불필요한 호출 제거:** 스크립트 안에서 Redis 명령 호출 횟수를 줄인다. 같은 키를 여러 번 읽을 필요가 있으면 **로컬 변수**에 담아 재사용한다. 불필요한 GET/SET을 반복하지 않는다.
- **의미:** 호출이 적을수록 스크립트 실행 시간이 짧아지고, 다른 클라이언트가 대기하는 시간도 줄어든다.

### 2. 테이블 효율

- **메모리 사용:** Lua **테이블**을 많이 쓰면 서버 메모리를 사용한다. 큰 테이블을 만들지 않고, 필요한 크기만 쓰거나, 가능하면 재사용·조기 해제를 고려한다.
- **의미:** 스크립트가 길거나 데이터가 크면 메모리와 실행 시간에 영향을 주므로, 테이블 사용을 필요한 범위로 제한하는 것이 좋다.

### 3. 조기 반환

- **불필요한 계산 방지:** 조건이 만족되면 **즉시 return**하고 나머지 로직을 실행하지 않는다. 예: 재고가 부족하면 `return -1`로 끝내고, 이후 SET 등을 하지 않는다.
- **의미:** CPU 사용과 실행 시간을 줄일 수 있다. 특히 에러·예외 케이스를 앞에서 걸러 주면 유리하다.

### 4. 캐시 활용 — EVALSHA 사용

- **EVALSHA 사용:** 같은 스크립트를 여러 번 실행할 때는 **SCRIPT LOAD**로 한 번 올린 뒤 **EVALSHA**로만 호출한다. 스크립트 문자열을 매번 보내지 않아 **네트워크와 파싱 비용**이 줄어든다.
- **의미:** 자주 쓰는 스크립트(재고 차감, rate limit, 복잡한 조건 갱신 등)에 적용하면 효과가 크다.

## redis.call vs redis.pcall

- **redis.call(...):** Redis 명령을 호출한다. 해당 명령이 **오류**를 반환하면 Lua 스크립트 전체가 **오류**로 끝나고, EVAL 호출자에게 오류가 전달된다.
- **redis.pcall(...):** 같은 방식으로 호출하되, Redis가 오류를 반환하면 **Lua 예외**로 변환되어 스크립트 안에서 `pcall`로 감싸 처리할 수 있다. “실패해도 스크립트는 계속 진행”하고 싶을 때 사용한다.

---

## 사용 시 유의사항

- **스크립트 실행 시간:** Lua 스크립트가 길게 돌면 그동안 Redis가 **다른 명령을 처리하지 못한다**. 무거운 연산이나 과도한 반복은 피하고, 가능하면 짧게 끝나도록 작성한다.
- **EVALSHA 실패 시:** 스크립트가 캐시에 없으면 EVALSHA는 오류를 반환한다. 클라이언트는 이때 **EVAL**로 스크립트 본문을 다시 보내서 실행하거나, SCRIPT LOAD 후 EVALSHA를 재시도하는 **폴백** 로직을 두는 것이 좋다.
- **키·인자 구분:** EVAL/EVALSHA의 **KEYS**와 **ARGV**를 구분해 쓴다. 키는 클러스터 샤딩 등에 사용되므로, 실제 키만 KEYS에 넣고 나머지는 ARGV에 넣는 것이 안전하다.
