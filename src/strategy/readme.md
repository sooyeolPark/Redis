# Redis 캐싱 전략

Redis를 **캐시**로 쓸 때, 데이터를 **언제·어떻게** 캐시에 넣고 DB와 맞추는지에 따라 여러 **캐싱 전략**이 있다. 읽기·쓰기 흐름과 일관성·성능·손실 위험을 이해하고 선택할 수 있도록 정리한다.

---

## 캐싱 전략 개요

| 전략 | 별칭/특성 | 핵심 장점 |
|------|-----------|-----------|
| **Cache-Aside** | 지연 로딩 (Lazy Loading) | **애플리케이션 제어.** 앱이 캐시를 확인하고, 없을 때만 DB에서 읽어 캐시에 넣는다. |
| **Write-Through** | 동기 쓰기 (Synchronous Write) | **강한 일관성.** 쓰기 시 캐시와 DB를 동시에 갱신해 항상 맞춘다. |
| **Write-Behind** | 비동기 쓰기 (Asynchronous Write) | **높은 쓰기 성능.** 캐시에만 먼저 쓰고, DB는 비동기로 나중에 반영한다. |
| **Read-Through** | 자동 로딩 (Automatic Loading) | **투명한 캐싱.** 앱은 캐시만 요청하고, 캐시 미스 시 캐시 레이어가 DB에서 읽어 채운 뒤 반환한다. |

---

## Cache-Aside (지연 로딩)

**애플리케이션**이 캐시(Redis)와 DB를 모두 알고 있고, **읽기 시** 캐시를 먼저 보는 패턴이다.

### 읽기 흐름 (캐시 미스 시)

1. **애플리케이션**이 **GET key**로 Redis에 조회한다.
2. **Redis**에 키가 없으면 **Cache Miss**를 반환한다.
3. **애플리케이션**이 **DB에 SELECT** 등으로 데이터를 조회한다.
4. DB가 **데이터**를 반환한다.
5. **애플리케이션**이 **SET key value**로 Redis에 저장한다. (필요하면 TTL 설정)
6. **애플리케이션**이 클라이언트에게 **데이터를 반환**한다.

**다음 요청**부터는 같은 키에 대해 Redis에서 바로 반환되므로 **캐시 히트**가 된다. DB를 거치지 않아 읽기가 빨라진다.

### 쓰기 시

- 보통 **DB를 먼저 갱신**한 뒤, 해당 키를 캐시에서 **삭제**(DEL)하거나 값을 **갱신**(SET)한다. 다음 읽기에서 캐시 미스가 나면 DB에서 최신 값을 가져와 다시 캐시에 넣는다.

### 특징

- **애플리케이션 제어:** “캐시에 넣을지, 언제 넣을지”를 앱이 결정한다. Redis는 단순 저장소 역할만 한다.
- **지연 로딩:** 요청이 온 키만 캐시에 올라가므로, 자주 쓰는 데이터만 캐시를 차지한다.

### 장단점

| 구분 | 내용 |
|------|------|
| **장점** | 구현이 단순하고, Redis·DB 모두 앱에서 제어할 수 있어 디버깅과 튜닝이 쉽다. 요청이 있는 키만 캐시에 올리므로 **메모리 효율**이 좋다. 캐시와 DB를 분리해 두어 **캐시 장애 시 DB만으로** 서비스가 가능하다. |
| **단점** | **캐시 미스 시** 첫 요청은 DB를 거치므로 지연이 길어지고, 동시에 많은 미스가 나면 **Cache Stampede**로 DB 부하가 폭주할 수 있다. 쓰기 시 "DB 갱신 후 캐시 무효화"를 앱이 직접 구현해야 하며, 그 순서를 잘못하면 **일시적 불일치**가 생길 수 있다. |

## Read-Through (자동 로딩)

**애플리케이션**은 **캐시 레이어만** 요청한다. 캐시에 없으면 **캐시 레이어(또는 캐시 라이브러리)**가 DB에서 읽어서 캐시에 넣은 뒤 반환한다.

### 읽기 흐름

1. **애플리케이션**이 **GET key** 등으로 **캐시 레이어**에만 요청한다.
2. **캐시 레이어**가 해당 키가 있는지 **캐시 확인**한다.
3. **캐시 미스**이면 캐시 레이어가 **DB에 SELECT**로 데이터를 조회하고, 받은 **데이터를 캐시에 저장**한 뒤 애플리케이션에 **데이터를 반환**한다.
4. **캐시 히트**이면 캐시에서 바로 **데이터를 반환**한다.

### 장단점

| 구분 | 내용 |
|------|------|
| **장점** | 앱은 **캐시만 호출**하면 되어 코드가 단순해지고, 캐시·DB 경계가 한곳(캐시 레이어)에 모여 **일관된 로딩 정책**을 적용하기 쉽다. 캐시 라이브러리/프록시를 쓰면 **재사용**과 **테스트**가 편하다. |
| **단점** | Read-Through를 지원하는 **캐시 레이어나 라이브러리**가 필요하다. Redis만으로는 구현되지 않으므로 앱 또는 별도 컴포넌트에서 미스 시 DB 조회 후 캐시 SET 로직을 구현해야 한다. **캐시 미스 시 첫 요청 지연**과 **동시 미스 시 Stampede** 위험이 있다. |

### 특징

- **투명한 캐싱:** 애플리케이션은 “캐시만 상대”하면 되고, DB 접근은 캐시 레이어(또는 라이브러리)가 담당한다.
- **자동 로딩:** 미스 시 캐시가 스스로 DB에서 채우므로, Cache-Aside와 비슷한 효과를 “캐시 추상화” 안에서 얻는다. Redis 단독으로는 이 로직이 없으므로, 앱 또는 캐시 프록시/라이브러리에서 구현한다.

## Write-Through (동기 쓰기)

쓰기 요청이 오면 **캐시와 DB를 모두 갱신한 뒤** 응답하는 패턴이다.

### 쓰기 흐름

1. **애플리케이션**이 **SET key value**로 Redis(캐시)에 쓴다.
2. **캐시 레이어**가 쓰기를 받으면 **동기적으로** DB에 **INSERT/UPDATE**를 수행한다.
3. **DB**가 성공을 반환하면 **캐시**가 그제서야 애플리케이션에 **완료**를 반환한다.

### 장단점

| 구분 | 내용 |
|------|------|
| **장점** | 쓰기 직후 **캐시와 DB가 항상 일치**하여 읽기에서 **오래된 값**을 보는 일이 없다. 장애 복구 시에도 캐시·DB가 같은 상태를 유지한다. **일관성**을 우선할 때 선택하기 좋다. |
| **단점** | **쓰기 지연**이 DB 쓰기 시간만큼 길어지고, 쓰기 요청이 많을수록 **DB 부하**가 커진다. 쓰기 처리량이 DB 성능에 묶인다. 캐시와 DB를 동시에 갱신하는 **앱/레이어 로직**이 필요하다. |

### 특징

- **캐시와 DB가 항상 동기화됨:** 쓰기 완료 시점에 캐시와 DB가 같은 값을 가진다. **강한 일관성**에 가깝다.
- **쓰기 지연:** 매 쓰기마다 DB 쓰기가 끝날 때까지 기다리므로, 쓰기 지연과 DB 부하는 커질 수 있다. Redis 단독으로는 “캐시가 DB에 쓰기”를 하지 않으므로, 이 동작은 애플리케이션 또는 중간 레이어에서 구현한다.

## Write-Behind (비동기 쓰기)

쓰기는 **캐시에만 먼저** 반영하고, **DB 쓰기는 비동기**로 나중에 처리하는 패턴이다.

### 쓰기 흐름

1. **애플리케이션**이 **SET key value**로 Redis에 쓴다.
2. **Redis**가 쓰기를 받으면 **즉시 완료**를 애플리케이션에 반환한다. (DB를 기다리지 않음)
3. **백그라운드**에서 Redis(또는 별도 워커)가 쓰기 내용을 **큐(메시지 큐, Redis Stream 등)**에 넣거나, 워커가 주기적으로 캐시 변경분을 DB에 **비동기로 INSERT/UPDATE**한다.
4. **나중에** 큐를 소비하는 워커가 DB에 실제로 반영한다.

### 장단점

| 구분 | 내용 |
|------|------|
| **장점** | **쓰기 지연이 짧고** 쓰기 처리량이 크게 늘어난다. DB 쓰기 부하를 **백그라운드**로 넘겨 DB 장애·지연이 쓰기 API에 바로 전파되지 않는다. 로그·이벤트·조회수처럼 **일시적 손실을 허용**할 수 있는 데이터에 적합하다. |
| **단점** | **데이터 손실 위험**이 있다. 캐시·큐·워커 장애 시 아직 DB에 반영되지 않은 쓰기가 사라질 수 있다. **캐시와 DB가 일시적으로 불일치**하므로, 쓰기 직후 읽기에서 최신 값이 안 나올 수 있다. 큐·워커·재시도 로직 등 **구현·운영 복잡도**가 올라간다. |

### 특징

- **쓰기가 매우 빠름:** 앱은 캐시에만 쓰고 끝나므로 쓰기 지연이 짧고 처리량이 높다. **높은 성능**을 노릴 수 있다.
- **데이터 손실 위험:** 캐시나 큐에만 있고 DB에 아직 반영되지 않은 시점에 **Redis·큐·워커가 죽으면** 해당 쓰기는 유실될 수 있다. “최종 일관성”과 “손실 가능성”을 감수하는 전략이다.

## Cache Stampede (캐시 스탬피드)

**캐시가 비었거나 키가 만료된 직후**, 같은 키를 **많은 클라이언트가 동시에** 요청할 때 발생하는 문제이다.

### 상황

- **캐시 만료 직후** 또는 **키가 없는 상태**에서, **클라이언트 1, 2, 3, …** 이 동시에 **GET key**를 Redis에 보낸다.
- Redis는 모두 **Cache Miss**를 반환한다.
- 각 클라이언트가 **동시에** DB에 **SELECT *** (같은 쿼리)를 보낸다. **중복 요청**이 한꺼번에 DB로 몰린다.
- 그 결과 **DB에 과부하**가 발생한다.

### 대응

- **락(Lock):** 캐시 미스 시 “이 키를 채우는 권한”을 한 클라이언트만 갖도록 분산 락(SETNX, Lua 등)을 건다. 한 클라이언트만 DB를 조회하고 캐시를 채운 뒤, 나머지는 캐시가 채워질 때까지 대기하거나 재조회한다.
- **Probabilistic Early Expiration:** TTL 직전에 일부 요청에서만 캐시를 미리 갱신해, 만료 시점에 한꺼번에 미스가 나는 것을 줄인다.
- **Stale-While-Revalidate:** 만료된 값이라도 일단 반환하고, 백그라운드에서 캐시를 갱신한다. DB 동시 요청을 줄이는 데 도움이 된다.

### 대응 방법별 장단점

| 방법 | 장점 | 단점 |
|------|------|------|
| **락** | 동시에 DB를 치는 요청을 **한 번으로** 줄여 Stampede를 확실히 막을 수 있다. | 락 대기로 **지연**이 생기고, 락을 잡은 클라이언트가 죽으면 **타임아웃**까지 다른 요청이 대기할 수 있다. 분산 락 구현·운영이 필요하다. |
| **Probabilistic Early Expiration** | 만료 시점에 미스가 몰리는 것을 **완화**하여 DB 부하를 분산한다. 락 없이 동작할 수 있다. | 완전히 제거되는 것은 아니며, TTL·확률 설정에 따라 효과가 달라진다. |
| **Stale-While-Revalidate** | 사용자에게 **즉시**(만료된 값이라도) 응답할 수 있어 체감 지연이 적다. DB 동시 요청을 줄인다. | **일시적으로 오래된 값**을 반환하므로, 최신성이 중요한 데이터에는 부적합할 수 있다. |

## 전략 선택 요약

| 목표 | 적합한 전략 |
|------|-------------|
| 읽기 위주, 앱이 캐시·DB 모두 제어 | **Cache-Aside** |
| 앱은 캐시만 보고, 미스 시 자동 로딩 | **Read-Through** (캐시 레이어/라이브러리 구현) |
| 쓰기 후 캐시·DB 항상 일치 필요 | **Write-Through** |
| 쓰기 처리량·지연이 최우선, 일시적 손실 허용 | **Write-Behind** |
| 캐시 만료 시 DB 동시 요청 폭주 방지 | **락, Early Expiration, Stale-While-Revalidate** 등 |

Redis는 위 전략들에서 **캐시 저장소** 역할을 하며, “캐시 미스 시 DB 조회 후 SET”, “쓰기 시 DB 동기/비동기 반영”, “Read-Through/Write-Through 로직”은 **애플리케이션 또는 캐시 레이어**에서 구현한다.
